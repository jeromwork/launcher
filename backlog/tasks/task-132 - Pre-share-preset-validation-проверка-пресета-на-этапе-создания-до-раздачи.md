## Что это простыми словами

Прежде чем пресет уходит людям (публикуется, шарится, кладётся в bundled-каталог), независимый механизм должен проверить его целиком и показать ошибки АВТОРУ пресета — а не пользователю после установки:
1. Автор собрал пресет (руками в JSON или в будущем UI-конструкторе).
2. Валидатор прогоняет проверки: все ссылки на пул существуют; компоненты, требующие других компонентов (`requires`), стоят в правильном порядке; нет циклов; обязательные шаги присутствуют; capability-требования объявлены.
3. Ошибки показываются сразу, до раздачи. Красный пресет раздать нельзя.

## Зачем

Слова владельца (2026-07-16): «эти ошибки выдали нам ещё на этапе создания пресета, а не после того, как мы его разослали пользователям». Сломанный пресет у пожилого пользователя = сломанный первый запуск.

## Что входит технически (для AI-агента)

- **Ядро уже существует**: [ValidationError.kt](../../core/src/commonMain/kotlin/com/launcher/preset/model/ValidationError.kt) — `UnknownPoolRef`, `RequiresOrderViolation`, `CircularOrdering`, `CapabilityMissing`, `SchemaVersionUnsupported` + validator в engine (TASK-120/126). Задача — не написать валидатор, а **обвязать его как authoring-gate**:
  - CLI/gradle-задача: `validatePresets` — прогоняет все bundled пресеты против пула на каждом build (fitness function per rule 7).
  - Тот же прогон как预 pre-share гейт в будущем UI-конструкторе пресетов.
  - Отчёт: локализуемые сообщения (у ValidationError уже есть `toI18nKey()`).
- Дополнительные проверки: непустые `tags`, валидность `layoutKey`, платформенная доступность (компонент без провайдера ни на одной платформе = предупреждение).

## Состояние
