## Что это простыми словами

Часы на телефоне пользователь может перевести. Обычному коду это безразлично, а криптографии — нет: если срок действия чего-либо проверяется по времени устройства, злоумышленник просто отматывает часы назад и получает доступ к тому, что уже должно было протухнуть.

**Что происходит при атаке:**
1. Сессия (или ключ, или приглашение) выдана со сроком «действует до 10:00».
2. Наступило 12:00 — она должна быть отвергнута.
3. Злоумышленник с доступом к устройству переводит системные часы на 09:00.
4. Код проверяет «сейчас раньше 10:00» по часам устройства, получает «да» и принимает просроченное.

Тот же приём работает и в другую сторону: подделав время, можно вставить в групповую переписку старую операцию так, будто она свежая.

**Защита:** в криптографическом коде время берётся только из источника, который контролирует сервер (`serverTimestamp` в Firestore или его аналог), а не из часов устройства. Это правило легко нарушить по невнимательности — `Clock.System.now()` пишется на автомате, — поэтому проверку ставим машинную, а не «помним и следим на ревью».

**Как работает проверка:** обычный юнит-тест обходит исходники и, найдя обращение к системным часам в криптографических пакетах, падает с объяснением. Инфраструктура уже есть — `ArchitectureFitnessTest` с четырьмя работающими проверками, каждая по 15–25 строк.

## Зачем

Закрывает класс атак на подмену времени (time-skew) там, где он действительно опасен. Без машинной проверки правило держится на памяти автора кода и внимательности ревьюера — то есть не держится.

## Что входит технически (для AI-агента)

**Правило:**
- Пакеты под надзором: `core/keys/*`, `core/crypto/*`, `app/adapters/openmls/*`, любой `*.pairing.*`.
- Запрещены: `kotlinx.datetime.Clock.System`, `java.lang.System.currentTimeMillis()`, `System.nanoTime()`.
- Escape hatch: `@Suppress("CryptoClockHygiene")` с обязательным комментарием-обоснованием (отладочные логи, некриптографическая инструментация).

**Реализация:** ещё один тест-метод в [`ArchitectureFitnessTest`](../../app/src/test/java/com/launcher/app/fitness/ArchitectureFitnessTest.kt) по образцу четырёх существующих. **Не** custom Detekt rule: модуль `lint-rules` удалён в TASK-140 — Detekt никогда не загружал наши правила, и они молча ничего не проверяли. Запуск: `./gradlew fitnessCheck`.

**Перед реализацией — аудит фактического состояния:** сначала grep по перечисленным пакетам. Если системные часы там уже используются, каждое место разбирается отдельно (перевод на server-authoritative источник либо обоснованный `@Suppress`). Правило включается только после того, как база чистая — иначе его отключат первым же коммитом.
