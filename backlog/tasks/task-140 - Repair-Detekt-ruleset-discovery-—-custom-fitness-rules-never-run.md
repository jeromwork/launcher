## Что это простыми словами

Фитнес-правила — это автоматические проверки, которые ломают сборку, если код нарушает архитектурную договорённость. У нас их четыре штуки в модуле `lint-rules/`, они написаны, покрыты тестами и, судя по документации проекта, «работают с TASK-65».

**Они не работают.** Ни одно из них ни разу не проверило ни строчки настоящего кода.

**Как это обнаружилось (2026-07-20, в ходе TASK-16):**
1. Написали четвёртое правило, прогнали по кодовой базе — ноль замечаний.
2. Это показалось подозрительным: в `SessionRecord.kt:21` лежит ровно тот случай, который правило обязано ловить.
3. Временно заставили правило сообщать о каждом файле подряд — снова ноль. Значит правило вообще не запускается.
4. Включили проверку конфигурации Detekt — он ответил: «свойство `launcher` не существует». То есть Detekt не знает про наш набор правил и никогда его не загружал.

**Почему так вышло.** Detekt находит внешние правила через механизм Java `ServiceLoader`: в jar-файле должен лежать служебный файл с именем класса-провайдера. Этого файла в `lint-rules/` не было вообще — три правила от TASK-65 были написаны, но не зарегистрированы. Правила компилировались, их собственные юнит-тесты проходили, никто ничего не заметил.

**Что уже сделано в TASK-16:** добавлен класс-провайдер `LauncherRuleSetProvider` и служебный файл регистрации, добавлен файл конфигурации `config/detekt.yml`. Проверено: jar собирается, служебный файл внутри лежит, jar доходит до Detekt. **Но набор всё равно не загружается, причина не найдена.** Раскопки остановлены, чтобы не уводить TASK-16 в сторону.

## Зачем

Пока это не починено, **все архитектурные проверки в проекте декоративны**. Зелёный `detektFoundation` означает «правила не запускались», а не «код в порядке» — то есть даёт ложную уверенность, что хуже отсутствия проверки.

Плюс это блокирует смысл соседних задач: TASK-16 (`wire-format-hygiene`) и TASK-139 (`crypto-flows-clock-hygiene`) пишут правила, которые не будут ничего проверять.

## Что входит технически (для AI-агента)

**Что проверено и исключено:**
- Служебный файл `META-INF/services/io.gitlab.arturbosch.detekt.api.RuleSetProvider` присутствует в собранном jar (проверено `unzip -l`).
- Jar попадает на plugin-classpath Detekt (виден в выводе `--info`).
- `detektPlugins` резолвится в `project :lint-rules` (проверено `:core:dependencies`).
- Версии байткода совпадают: Gradle JVM 21, class major version 65 — `UnsupportedClassVersionError` исключён.
- Принудительный выбор jar-варианта через `LibraryElements.JAR` не помог (изменение откачено как бесполезное).
- Ключи правил в конфиге исправлены на имена классов (Detekt резолвит через `Rule.ruleId` = simple class name, а не через `issue.id`) — не помогло, потому что не грузится сам набор.

**Что проверить дальше:**
- Совместимость Detekt 1.23.7 (собран под Kotlin 1.9) с модулем правил, скомпилированным Kotlin 2.0.21 — наиболее вероятный кандидат.
- Фактический `toolVersion` расширения Detekt против `libs.detekt.api` (1.23.7).
- Не проглатывается ли `ServiceConfigurationError` молча — запустить Detekt CLI напрямую с `--plugins`, минуя Gradle, чтобы увидеть настоящую ошибку.
- Вариант обхода: перенести правила в `Konsist`-тесты (уже подключён в `core/`, `app/`, `core/push/`) — они запускаются как обычные тесты и не зависят от механизма загрузки Detekt.

**Файлы:** [`lint-rules/`](../../lint-rules/), [`config/detekt.yml`](../../config/detekt.yml), [`build.gradle.kts`](../../build.gradle.kts) (блок `subprojects` + задача `detektFoundation`).

**Побочный факт, зафиксировать:** до 2026-07-20 задача `detektFoundation` падала с 556 замечаниями стандартных правил Detekt (`MaxLineLength`, `MagicNumber`) — `buildUponDefaultConfig = false` без файла конфигурации не отключает стандартный набор, как предполагалось. Файл `config/detekt.yml`, добавленный в TASK-16, это чинит: стандартные наборы отключены явно.

## Состояние

**Done** (2026-07-20). Правила перенесены из Detekt в обычные юнит-тесты и впервые за всё время реально прогнаны по кодовой базе.

**Почему не чинили Detekt.** Решающий аргумент — не «Detekt сломан», а **молчание**: правило, которое не загрузилось, ничего не сообщает и считается пройденным. Именно поэтому поломку не замечали с TASK-65. Юнит-тест так не умеет: если файл есть, Gradle его запустит, а переставшее срабатывать правило станет красным тестом, а не тишиной. Плюс запасной путь (Konsist / обычные тесты) был заранее записан в эту же задачу, а Konsist уже использовался в проекте — в комментарии соседнего теста прямо сказано, что он выбран как «путь без custom Detekt plugin».

**Что удалено:** модуль `lint-rules/` (4 детектора + провайдер + тесты), `config/detekt.yml`, плагин Detekt из сборки, записи в version catalog. Оставлять неработающий механизм было опасно: следующий разработчик добавил бы туда пятое правило, и оно так же молча не запустилось бы.

**Замена:** [`ArchitectureFitnessTest`](../../app/src/test/java/com/launcher/app/fitness/ArchitectureFitnessTest.kt) — те же четыре правила, запуск `./gradlew fitnessCheck`.
