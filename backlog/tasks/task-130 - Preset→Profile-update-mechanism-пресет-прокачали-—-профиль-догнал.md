## Что это простыми словами

Пресет (шаблон настроек) со временем улучшается — выходит его новая версия (`version: 2 → 3`). Профиль на устройстве был скопирован из старой версии. Нужен механизм «профиль догоняет пресет»:
1. Приложение видит: `Profile.presetVersion` < версии пресета в каталоге.
2. Считает разницу (что добавилось/изменилось/удалилось в пресете).
3. Аккуратно вливает изменения в профиль, НЕ затирая правки пользователя (если пользователь сам менял размер шрифта — обновление пресета его выбор не трогает).
4. Новые компоненты проходят через движок применения (ReconcileEngine) как обычно.

Edge case: пресет изменил то же поле, что и пользователь — нужна политика конфликта (по умолчанию: выбор пользователя сильнее).

## Зачем

Прокачка пресетов заявлена как основной сценарий развития («улучшаем пресеты и раздаём»). Без этого механизма каждый установленный профиль навсегда замораживает старую версию пресета, и улучшения доходят только до новых установок.

## Что входит технически (для AI-агента)

- Diff по `basedOnPreset` + `presetVersion` (швы уже в формате: [Profile.kt](../../core/src/commonMain/kotlin/com/launcher/preset/model/Profile.kt) — `basedOnPreset`, `presetVersion`, `unknownRefs`).
- Merge policy: preset-default vs user-override vs conflict (user wins по умолчанию; политика — кандидат в preset-field per rule 11).
- Прогон новых компонентов через `ReconcileEngine` (возможно новый RunMode или reuse RemotePush-ветки).
- Wire-format не меняется (additive); roundtrip + сценарные тесты (обновление без правок / с правками / с конфликтом).
