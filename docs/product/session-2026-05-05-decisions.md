# Решения сессии 2026-05-05

Источник: восстановленный транскрипт `.recovered-session-2026-05-05-clean.md`.
Все решения приняты в ходе обсуждения с владельцем проекта.

## 1. Один APK против двух APK

**Решение:** один APK с двумя режимами (launcher / admin).

**Почему рассматривались альтернативы:**
Два APK с общим `:core` — стандартный паттерн для продуктов с сильно разными аудиториями. Позволяет иметь раздельные store-листинги и отдельную цену.

**Отклонённый вариант:** два APK.
Причина отклонения: усложняет поддержку, store-listings, pricing-логику (ADR-003). При едином APK оба режима переиспользуют один каркас UI и core-модели.

**К чему пришли:** один APK, роль определяется пресетом, выбранным на first-launch. Никакого runtime mode-switch в production. Dev-toggle (в hidden Settings) присутствует только в debug-сборке и убирается из release.

## 2. Backend: Firebase против альтернатив

**Решение:** Firebase Firestore + FCM для MVP, но за интерфейсом `RemoteSyncBackend` (не vendor lock-in).

**Почему рассматривались альтернативы:**
Remote-channel требует реального backend. Вопрос — какого.

**Рассмотренные варианты:**

| Вариант | Плюсы | Минусы | Решение |
|---|---|---|---|
| Firebase Firestore + FCM | Быстрый MVP, free tier (Spark), бесплатный FCM, готовый realtime-listener | Vendor lock-in, privacy/GDPR компромиссы | Выбран для MVP |
| Supabase | Open-source, self-host | Дороже в operations | Отклонён для MVP |
| Custom REST + WebSocket + FCM/APNs | Максимум контроля | Максимум работы | Отклонён для MVP |
| WebRTC P2P (как RustDesk) | Максимально privacy-friendly | Pairing-UX сложнее, offline-доставка плохая | Отклонён |

**К чему пришли:** Firebase выбран как MVP-реализация за интерфейсом абстракции. Смена провайдера в будущем требует замены только одного модуля.

## 3. Firebase как абстрактный RemoteSyncBackend (не vendor lock-in)

**Решение:** конституция (FR-013) запрещает remote-config в `:core`. Firebase — это adapter, не часть домена.

**Почему рассматривался вопрос:**
Владелец явно поставил требование: «Firebase лишь один из возможных вариантов. В будущем должно быть своё хранилище».

**Архитектурное решение:**
```
:core
  └── api/sync/RemoteSyncBackend.kt   ← интерфейс, без реализации
:feature-remote-firebase              ← Firebase impl (Spark-tier для MVP)
:feature-remote-fake                  ← in-memory impl для тестов и dev
:feature-remote-selfhost  (future)    ← будущая своя реализация
```

Контракт `RemoteSyncBackend`:
- `startPairingAsOld(): PairingArtifact`
- `completePairingAsAdmin(qr: String): PairingResult`
- `observeConfig(linkId): Flow<RemoteConfig>`
- `pushConfig(linkId, cfg)`
- `observeState(linkId): Flow<RemoteState>`
- `pushState(linkId, state)`
- `pushHealth(linkId, h: HealthSnapshot)`
- `unlink(linkId)`

DI выбирает реализацию при сборке. Vendor lock-in на уровне домена исключён.

## 4. Pairing UX: QR на OLD, admin сканирует

**Решение:** QR-код генерируется на телефоне OLD, admin сканирует своим телефоном.

**Почему рассматривались альтернативы:**
Два варианта: (а) QR на OLD, admin сканирует; (б) 6-значный код на OLD, admin читает по телефону / вводит.

**Отклонённый вариант:** числовой код.
Вариант QR выбран по двум причинам:
1. Безопасность — admin должен физически присутствовать рядом с OLD при первом подключении.
2. Предположение, что admin (человек средних лет) умеет сканировать QR.

**К чему пришли:**
- При первоначальной настройке или через Settings → «Разрешить удалённое управление» → «Показать QR-код».
- Реальный QR через ZXing (offline, Apache 2.0, без GMS-зависимости).
- В Этапе 0 (spec 003) — dummy-токен (`launcher://pair?token=dev-stub-UUID`), реальная логика в spec 006.

## 5. Privacy/consent на OLD при первом подключении

**Решение:** явный consent-экран на устройстве OLD при первом подключении к admin'у.

**Почему это необходимо:**
- Юридически обязательно (GDPR Art.7).
- Этически: пожилой человек должен знать, кто и что может видеть/менять на его телефоне.

**К чему пришли:**
- После того как admin отсканировал QR — OLD показывает consent-экран с перечнем: «Admin <name> сможет видеть и менять <список>». Кнопки «Разрешить» / «Отклонить».
- Пользователь подтверждает физически на OLD — без этого привязка не завершается.
- В spec 003 — заглушка (экран с текстом, без сети). Реальная логика в spec 006.

**Что сохраняется как идея:** warning-state, confirm-screen перед уходом во внешнее приложение, restore-on-return — это паттерны, переедут в универсальную provider-архитектуру.

## 7. Пресеты: не one-shot блокировка, а switchable UX-профили

**Решение:** пресет — это переключаемый профиль UX-режима, ортогональный пользовательским данным. Смена пресета не уничтожает конфигурацию.

**Почему рассматривалась «жёсткая блокировка»:**
Исходный замысел (PDF) — пресет как обязательный чек-лист системных настроек. Но пожилой/admin физически находится рядом при первой установке и может не пройти все шаги.

**Отклонённый вариант:** one-shot обязательный wizard, из которого нельзя выйти без выполнения всех требований.

**К чему пришли:** три переключаемых пресета:

| Пресет | Кому | Системные требования | Bottom-nav | Кнопки |
|---|---|---|---|---|
| `flow-light` | Обычный пользователь / телефон admin'а | Никаких | Всегда видна | Material density |
| `senior-launcher` | OLD-пользователь, мягкий режим | Default launcher (рекомендуется), battery-opt off (рекомендуется) | Скрыта при одном flow | Крупные (56dp+, 18sp+) |
| `senior-launcher-strict` | future — с полным контролем / DPC | Default launcher (обязательно), lock-task | Скрыта | Крупные |

**Механика переключения:**
1. Settings → «Сменить режим» → список пресетов с описанием.
2. При выборе — diff-проверка: что выполнено ✓, что не выполнено `!`.
3. Пользователь решает настроить сейчас или позже. Пресет применяется немедленно, требования — инкрементально.
4. В Settings всегда виден баннер «Активный пресет: X. Не выполнено: N требований».

## 8. «Управление другими телефонами» — flow template, не Settings

**Решение:** «Управление другими телефонами» — это flow template в bottom-nav, наравне с «Общение» и «Развлечения».

**Почему рассматривалось размещение в Settings:**
Логично, что настройка подключения — административная операция, а не ежедневная. Большинство систем (Family Link, Screen Time) прячут управление в Settings.

**Отклонённый вариант:** Settings → подраздел.

**Причина отклонения:** мониторинг состояния OLD — одна из killer-фич. Admin'у нужно видеть «как там у пожилого» быстро. Settings → раздел → подраздел = три тапа против одного таба. Это сломанный приоритет.

**К чему пришли:**
- «Управление другими телефонами» — отдельный flow template, доступен в picker'е при нажатии «+».
- Gate по пресету: виден только в `flow-light`, скрыт в `senior-launcher` (пожилой не должен случайно стать чьим-то admin'ом).
- В Settings остаётся только «Разрешить удалённое управление **этим** телефоном» (toggle + QR) — privacy-чувствительная настройка, делается редко.

## 9. Mode switch убран из production; роль = функция пресета

**Решение:** нет отдельного dev-переключателя «launcher mode / admin mode» в production.

**Почему рассматривался отдельный toggle:**
Удобство разработки — один APK, можно переключаться между режимами без переустановки.

**Отклонённый вариант:** runtime mode-switch в production.

**К чему пришли:**
- Роль определяется пресетом, выбранным на first-launch. Admin-режим = `flow-light` + flow «Управление другими телефонами». OLD = `senior-launcher`.
- В release нет ни single-bit'а переключения режима.
- Для разработки: два AVD одновременно, каждый с независимыми данными и независимо выбранным пресетом. Не нужно никаких debug-флагов.

## 10. ZXing-android-embedded для QR (offline, бесплатно, без GMS)

**Решение:** `com.journeyapps:zxing-android-embedded:4.3.0`.

**Рассмотренные варианты:**

| Библиотека | Offline | GMS | Вес | Рекомендация |
|---|---|---|---|---|
| `zxing-android-embedded` | Да | Не нужен | ~1 MB | Выбрана для MVP |
| `zxing:core` | Да | Не нужен | ~600 KB | Только если нужен только генератор |
| ML Kit Barcode Scanning | Да (runtime-модель) | Нужен + подгрузка ~2-3 MB | Мал | Избыточен на Этапе 0 |

**К чему пришли:** ZXing-android-embedded выбран — один gradle-стринг, генерация и сканер из коробки, без GMS-зависимости (работает на устройствах без Google-сервисов), Apache 2.0, бесплатно навсегда.

## 11. Иконки — собственные, не бренд-логотипы

**Решение:** все провайдер-иконки — собственный дизайн, никаких загрузок логотипов WhatsApp/Telegram/YouTube из их брендбуков.

**Почему рассматривались брендированные иконки:**
Все senior-launcher'ы на рынке используют цветные брендированные логотипы — пожилым они узнаваемее.

**К чему пришли:** собственные SVG-иконки, узнаваемые по силуэту. Цвет несёт смысловую нагрузку (зелёный = звонок, красный = экстренное, серый = нейтральное), не принадлежность к бренду.

В Этапе 0 (spec 003) — плейсхолдеры из Material Icons (Apache 2.0) с TODO «заменить на custom set» в spec 004. Это two-way door.

## 12. Persistence: DataStore вместо SharedPreferences

**Решение:** `DataStore` (Jetpack, Google) с первого дня.

**Рассмотренные варианты:** `SharedPreferences` vs `DataStore`. Google deprecated рекомендует DataStore.

**К чему пришли:** DataStore — типизированный, асинхронный, миграции встроены. Не дороже SharedPreferences в реализации, но future-proof. Room понадобится только в spec 007 при синхронизации/conflict resolution.

**Решение:** ручная конструктор-инъекция через `LauncherApplication`. Hilt/Dagger — запрещены в MVP конституцией.

## 15. CLAUDE.md как место для незыблемых правил

**Решение:** `CLAUDE.md` в корне репозитория содержит только незыблемые engineering-правила. Никаких ссылок на конкретные файлы проекта, только принципы.

**Почему рассматривались альтернативы:**
- Skills (Claude Code) — отклонены: они для узких операций, не для постоянных правил.
- Конституция — уже существует, `CLAUDE.md` не дублирует, а дополняет.
- Полная копия конституции в `CLAUDE.md` — отклонена: слишком большой контекст в каждом промпте.

**К чему пришли:** `CLAUDE.md` — короткий (~78 строк) entry-point, содержит 7 незыблемых правил, которых не было в конституции (Hexagonal, ACL, one-way doors, MVA, wire-format versioning, mock-first, fitness functions), и 11 anti-patterns. Конституция остаётся единственным источником правды.

**Рассмотренные варианты:**

- Один AVD + физический телефон.

## 18. Config portability при замене телефона — backlog

**Решение:** зафиксировать как backlog, не блокирует текущие spec'и.

**Что зафиксировано:**
Отдельный stub-файл `specs/backlog/config-portability.md`. При замене OLD-устройства admin должен иметь возможность перепривязать новый телефон к существующему `linkId`/конфигу.

Варианты (для будущего spec'а):
1. Bind по `linkId` в облаке: admin выпускает новый QR для существующего устройства.
2. Local export/import: конфиг в файл → импорт на новом телефоне.
3. Cloud backup snapshot: автосохранение слотов на дату.
