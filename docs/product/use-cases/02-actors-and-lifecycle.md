# 02. Actors & Lifecycle — кто наши пользователи и их стадии жизни с продуктом

> **Status**: 🆕 первый проход · **Created**: 2026-05-27
> **Зачем читать**: каждая фича в продукте отвечает «для какого актора на какой стадии». Без этого ответа фича либо «для всех» (плохо), либо «для никого» (ещё хуже).
> **Источник**: `user-journeys-draft.md` §3.

---

## Что это за документ (просто)

Когда говорят «пользователь», то в нашем продукте это **не один человек**. Тут как минимум двое: тот, кто пользуется лаунчером (Managed), и тот, кто настраивает (Admin). И они **на разных устройствах**, **с разным опытом**, и часто **в разных городах**.

Этот документ перечисляет всех акторов (типичных «персонажей» продукта) и проходит по их **стадиям жизни с продуктом**:
- как они узнают (хотя discovery мы вынесли в другой проект — см. user-journeys-draft.md §3.0),
- как настраивают в первый раз,
- как пользуются ежедневно,
- что меняют со временем,
- что происходит при поломке / потере / смене устройства,
- что в конце (end-of-life).

Без этого инвентаря фичи получаются «удобные для разработчика», а не «удобные для конкретного человека на конкретной стадии».

## Главные понятия (просто)

- **Актор (actor)** — собирательный образ пользователя, у которого свои цели, контекст, ограничения. Не «один реальный человек», а **роль**. Один реальный человек может быть в нескольких ролях (например, admin для бабушки + Managed на отдельном тестовом телефоне).
- **Persona vs actor** — persona глубже, со своими страхами и био. Мы пока обходимся actor'ами — короче, проще.
- **Lifecycle stage (стадия жизни)** — где пользователь находится сейчас относительно продукта: только установил / пользуется месяц / меняет телефон / удалил. Каждая стадия — свой набор фич, свой UX.
- **Happy path / sad path / edge case** — happy = всё идёт хорошо; sad = что-то частично не работает; edge = редко, но критично (потеря, кража, factory reset).

## Список акторов

### A1. Managed user (главный потребитель)

**Кто это**: пожилой человек, обычно 65+, иногда с одним из ограничений: снижение зрения, слух, тремор, когнитив. Это **главный пользователь устройства**, на котором стоит наш лаунчер. Цель — общаться с семьёй простыми способами (звонки, видео, фото) и не пугаться телефона.

**Характеристики**:
- Часто **не первый раз** видит смартфон, но не уверен в нём.
- Боится «сломать» — поэтому редко экспериментирует.
- Контекстная память на UI слабая («где была эта кнопка?»). Любит **стабильность** размещения.
- Зрение, моторика, когнитив могут ухудшаться **со временем** — UI должен быть устойчив к этому.
- **Часто не использует email** или другие технические каналы связи.

**Не путать с**:
- «Just non-tech-savvy user» (50-летний, не любящий технологии) — это не наша главная аудитория, но boundary case.
- Ребёнок / подросток — у нас нет parental control flow, мы не для них.

### A2. Admin (типичная персона: внук / сын / дочь)

**Кто это**: близкий родственник Managed user'а, обычно 25-50 лет, технически грамотный, **на расстоянии** (другой город / другой район). Цель — настроить устройство бабушки и иногда поправить.

**Характеристики**:
- Использует **свой собственный** телефон (Android или iOS) как admin-устройство.
- Заходит в наше приложение **редко** — раз в неделю или реже. UI должен быть **узнаваемым после паузы**.
- Не хочет тратить часы — нужны **быстрые** действия.
- Часто **отвлекается** (на работе, в дороге) — flow должен переживать прерывания.
- Может быть **не один** — есть co-admin'ы (другие родственники).

### A3. Co-admin (другой родственник)

**Кто это**: второй (третий, ...) родственник, тоже настраивает то же Managed-устройство. Цель — координация с основным admin'ом, не дублировать работу.

**Характеристики**:
- Может быть **дальше или ближе** к Managed'у, чем основной admin.
- Часто **не такой технически активный** — например, муж дочери.
- **Не должен ломать** то, что сделал основной admin (или конфликт виден и решаем).

### A4. Bystander (помощник в моменте)

**Кто это**: кто-то рядом с Managed-устройством — сосед, медсестра, ситуативный помощник. **Не admin** — у него нет постоянного доступа, но в моменте может помочь нажать.

**Характеристики**:
- Не знает наш UI, помогает «на здравом смысле».
- UI должен быть **очевиден без обучения** — иначе не поможет.

### A5. Vendor / Support / Partner (будущее)

**Кто это**: внешний агент — клиника, retirement home, security-центр, hardware-датчик. Не human user, но «логический актор», который потребляет / посылает данные.

**Характеристики**:
- Доступ — через ограниченный канал (read-only health data, write-only emergency alert, etc.).
- **Не должен видеть** privacy data.
- Сейчас не реализован, но **закладывается архитектурно** в спеке 011 (OWD-5 per-app identity).

### A6. Сам разработчик / тестировщик (DX)

**Кто это**: мы. Цель — проверить фичу end-to-end. Это **отдельный актор**, потому что у нас **нет** возможности локально протестировать многое (см. S-105 / S-801).

**Характеристики**:
- Один человек, обычно один эмулятор или два рядом.
- Не хочет setup'ить два настоящих устройства для каждого теста.
- Нужен **локальный admin UI** или **headless admin CLI** — иначе DX страдает.

## Lifecycle stages — стадии жизни с продуктом

### L1. First-time setup (первичная настройка)

Кратко — открыли коробку / установили app, надо чтобы заработало. Самая критическая стадия — большинство пользователей **бросают** продукт именно здесь, если что-то идёт не так.

| ID | Кейс | Актор | Status |
|---|---|---|---|
| S-101 | Идеальный first-time (внук настроил → бабушке передали готовое) | Admin + Managed | 🟡 (007+010 частично) |
| S-102 | Setup в одной комнате (admin рядом с Managed) | Admin + Managed | 🟡 |
| S-103 | Setup remote (admin в другом городе) | Admin + Managed | ❌ |
| S-104 | Self-serve (нет admin'а) | Managed only | ❓ D-1 |
| S-105 | Setup тестировщиком / разработчиком | Dev | ❌ (boil) |
| S-106 | Setup в B2B (клиника / retirement home) | Partner | 🔮 |
| S-107 | Setup с миграции (был другой launcher, поставили наш) | Admin + Managed | ❌ |

### L2. Daily use (ежедневное использование)

Что Managed делает каждый день / неделю. Должно быть **скучно** и **предсказуемо** — то, что в этом стадии «удивляет», это плохой UX.

| ID | Кейс | Актор | Status |
|---|---|---|---|
| S-201 | Бабушка звонит внуку через тайл | Managed | ✅ (002) |
| S-202 | Бабушка нажимает SOS | Managed | ❌ |
| S-203 | Бабушка случайно открыла что-то лишнее | Managed | 🟡 |
| S-204 | Бабушка читает напоминание о таблетках | Managed | ❌ (D-4) |
| S-205 | Возврат из мессенджера в лаунчер | Managed | ✅ (002) |
| S-206 | Видит фото внука на плитке | Managed | 🟡 (012 stub) |
| S-207 | Видит «нет связи» | Managed | ❌ (deferred → 013) |
| S-208 | Managed показывает свой QR | Managed | ❌ |
| S-209 | Bystander помогает Managed нажать на тайл | Bystander | ❓ |

### L3. Re-configuration (плановые изменения)

Что admin делает периодически. **Должно быть лёгким** — иначе admin перестанет, и конфигурация со временем «осыпется».

| ID | Кейс | Актор | Status |
|---|---|---|---|
| S-301 | Admin добавляет новый контакт удалённо | Admin | ✅ (009) |
| S-302 | Admin добавляет co-admin'a | Admin | ❌ |
| S-303 | У внука сменился номер телефона | Admin | ❌ |
| S-304 | Admin меняет своё фото — у Managed обновляется | Admin → Managed | 🟡 (012 stub) |
| S-305 | Admin делает rollback на старый конфиг | Admin | ✅ (009) |
| S-306 | Admin / Managed меняет язык интерфейса | Managed | ❌ |
| S-307 | Admin меняет grid-size / text-size после первичной настройки | Admin | ❌ |

### L4. Recovery & failure (что-то сломалось)

Самая болевая стадия. Чем тяжелее actor'у в этой стадии — тем сильнее он либо переключится на конкурента, либо потеряет данные, либо подаст в support.

| ID | Кейс | Актор | Status |
|---|---|---|---|
| S-401 | Бабушка случайно сделала factory reset | Managed → Admin | ❌ (011 OWD-4 social recovery — будущая часть) |
| S-402 | Internet outage у Managed на неделю | Managed | 🟡 (deferred → 013) |
| S-403 | FCM не доставился (Doze / OEM kill) | Managed | ✅ (007 polling fallback) |
| S-404 | Admin потерял свой телефон | Admin | ❌ |
| S-405 | Cloudflare Worker недоступен | All | ❌ |
| S-406 | Firebase Spark лимиты превышены | All | ❌ |
| S-407 | Permission revoked (OS update / OEM clean) | Managed | ✅ (010) |
| S-408 | Малое место на устройстве / DB ошибка | Managed | ❌ |

### L5. Migration (смена устройства)

| ID | Кейс | Актор | Status |
|---|---|---|---|
| S-501 | Бабушка получает новый телефон | Managed + Admin | 🔮 (011 OWD-4) |
| S-502 | Admin меняет телефон | Admin | 🟡 (008 multi-device) |
| S-503 | App update с breaking wire-format | All | 🟡 (rule 5) |

| ID | Кейс | Актор | Status |

## Открытые вопросы (Actor / Lifecycle specific)

### D-1A. Поддерживаем ли мы actor A4 (Bystander) явно?

**Контекст**: «случайный помощник» — нигде не зафиксирован, но реально присутствует (медсестра, сосед).

**Варианты**:
- **Игнорируем** — UI делаем для admin + Managed, bystander «как-нибудь сам».
- **Учитываем как guideline** — UI должен быть очевиден без обучения. Это уже наш default.
- **Делаем явный bystander mode** — отдельный entry point «помощь без логина».

**Рекомендация**: вариант 2 (guideline). Не делать отдельного mode'а — это choice-overload. Просто проверять каждый flow на «понятно ли это случайному взрослому».

### D-2A. Существует ли actor A6 (Dev) официально?

**Контекст**: текущая боль user'а — нет local edit UI, тестировщик не может. Если **не признать** A6 как актора, то DX не улучшится.

**Варианты**:
- **Не признаём** — DX страдает, медленнее работаем, ошибки находятся пользователями вместо тестов.
- **Признаём** — добавляется требование: каждая user-facing спека имеет **local end-to-end test path** (debug build, локальный admin UI, headless tools).

**Рекомендация**: признать. Связано с D-2 в README (vertical-slice testing).

### D-3A. Lifecycle stage «end-of-life» — наша ответственность?

См. D-3 в README. Это про L6.

**Варианты**:
- **Не наша**: «семья сама разбирается», ключи / данные просто удаляются при factory reset.
- **Частично наша**: предоставляем механизм наследования (digital legacy: «передать admin-роль X-у»).
- **Полностью наша**: предусматриваем legal-grade процедуры (правопреемство, удаление по запросу третьих лиц, etc.).

**Регрет-условие**: если выберем «не наша», то reputational hit при первом громком случае («бабушка умерла, семья не может ничего достать»).

**Рекомендация (best-guess)**: «частично». Закладываем architectural support (через 011 OWD-4 social recovery) уже сейчас, отдельную UX-фичу — позже.

### D-4A. Lifecycle stage «migration» (L5) — priority

Сейчас migration — это **дыра**. У нас даже нет ясного flow «новая бабушкина симка / новый телефон».

**Регрет-условие**: первый же реальный пользователь, у которого сломался телефон, потеряет всю конфигурацию.

**Рекомендация**: вынести L5 в **MVP scope**. Сейчас не вынесено.
