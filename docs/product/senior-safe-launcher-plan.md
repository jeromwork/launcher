# Senior Safe Launcher: Product & Technical Plan

## 1. Цель

Сделать режим телефона, в котором пожилой пользователь видит один предсказуемый экран с разрешенными действиями и не может случайно:

- открыть лишние системные панели;
- перейти в нежелательные приложения;
- изменить критичные настройки;
- потеряться в навигации.

Ключевой UX-эффект: "телефон не пугает, потому что ведет себя всегда одинаково".

## 2. Что ожидают пользователи и помощники

Приоритеты (по рынку senior-launcher решений и caregiver-сценариям):

1. Простота вместо гибкости: крупные кнопки, минимум сценариев.
2. Предсказуемость: один главный экран, отсутствие "случайных" переходов.
3. Защита от ошибок: скрытие лишних функций и PIN-защита настроек.
4. Быстрые контакты и SOS.
5. Прозрачный setup для помощника: что уже выдано, что еще нужно.

## 3. Платформенные ограничения и модель режимов

Полный "lock-down" невозможен в consumer-режиме без системного управления устройством. Поэтому продукт строится в двух режимах:

1. Standard (consumer install): базовая защита и упрощенный UX без полного системного контроля.
2. Managed/Strict (управляемое устройство): kiosk/single-app политики и строгие ограничения.

### 3.1 Android

- Полноценный strict-режим требует Device Owner / Dedicated Device подхода.
- Только default launcher без DPC-политик не даст полного контроля кнопок/шторки/переключения задач.

### 3.2 iPhone

- На обычном iPhone нельзя заменить Home/Launcher.
- Для строгого режима нужен supervised device + MDM (App Lock / Single App Mode и supervised restrictions).
- Guided Access подходит как частичный/временный lock, но не как централизованный продуктовый аналог dedicated Android режима.

## 4. Целевое поведение strict-режима

1. Launcher — единственная "домашняя" точка.
2. Разрешен только whitelist приложений.
3. Шторка/уведомления/recents ограничены политиками.
4. Из внешнего мессенджера пользователь возвращается в launcher (через Home/back-task поведение ОС).
5. Служебные настройки доступны только помощнику через скрытый вход + PIN.

Важно: внутри сторонних приложений (Telegram/WhatsApp) полностью контролировать их внутренний UX нельзя.

## 5. Разрешения, роли и политики

### 5.1 Android (минимум для managed/strict)

1. Role: `ROLE_HOME` (launcher по умолчанию).
2. Device Owner / DPC provisioning.
3. Lock Task allowlist:
   - `setLockTaskPackages(...)`
   - `setLockTaskFeatures(...)`
4. User restrictions:
   - `DISALLOW_CREATE_WINDOWS`
   - `DISALLOW_ADJUST_VOLUME` (если подтверждаем продуктово)
5. Дополнительно по устройству:
   - уведомления, battery optimization, vendor-specific autostart пункты.

### 5.2 iPhone (аналог strict)

1. Supervision (через ADE/Configurator).
2. MDM enrollment.
3. App Lock / Single App Mode профиль.
4. Supervised restrictions (по матрице корпоративных ограничений).

## 6. Отдельный модуль Setup Assistant

Создать модуль `:setup-assistant` для централизованной выдачи и контроля доступа.

### 6.1 Задачи модуля

1. Показать статус всех требований: `Granted / Missing / Unsupported`.
2. Дать пошаговый сценарий выдачи (рекомендуется).
3. Поддержать "пакетную" проверку и повторную валидацию.
4. Давать понятные объяснения "зачем это нужно" для каждого пункта.

### 6.2 Архитектура (расширяемость)

Реестр требований (policy-driven):

- `id`
- `title`
- `description`
- `criticality` (`required`, `recommended`, `optional`)
- `platform` (`android`, `ios`, `both`)
- `check()` (факт наличия)
- `resolveIntent/deepLink` (куда вести настройщика)
- `telemetryKey`

Добавление нового разрешения в будущем = добавление одной записи в реестр + локализация.

## 7. Скрытый вход для помощника

1. Триггер: 7 тапов подряд по выделенной зоне.
2. Затем запрос PIN.
3. Константы (`tapCount`, `windowMs`) выносятся в конфиг.
4. В будущем допускается смена триггера без переписывания бизнес-логики.

## 8. Roadmap реализации

### Этап 1: Foundation

1. Вынести `:setup-assistant`.
2. Реестр требований + статусы + экран прогресса.
3. Скрытый вход 7 тапов + PIN.

### Этап 2: Android strict

1. DPC/provisioning поток.
2. Lock Task + whitelist приложений.
3. Ограничения системного UI и нежелательных действий.

### Этап 3: iPhone strict

1. Supervision/MDM deployment profile.
2. Единый чек-лист setup для помощника.
3. Платформенная матрица "что доступно/что нет".

### Этап 4: Supportability

1. Диагностический отчет "чего не хватает до full protection".
2. Экспорт отчета для техподдержки/помощника.
3. Регресс-набор по вендорам Android и iOS supervision-сценариям.

## 9. Риски и ограничения

1. Невозможность полного контроля в consumer-режиме.
2. Различия OEM-прошивок Android (Samsung/Xiaomi/Realme и т.д.).
3. На iOS строгий режим требует инфраструктуры MDM/supervision.
4. Юридические и UX-ограничения при агрессивной блокировке системных действий.

## 10. Definition of Done (MVP strict)

1. Устройство загружается в launcher как единственную рабочую поверхность.
2. Доступны только разрешенные действия из whitelist.
3. Setup Assistant показывает 100%/N% готовность с точной причиной каждого missing пункта.
4. Вход в служебный экран возможен только через скрытый trigger + PIN.
5. Добавление нового разрешения не требует правки экранной логики (только реестр + тексты).
