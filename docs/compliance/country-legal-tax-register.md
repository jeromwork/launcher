# Country Legal and Tax Register

## Цель
Держать живой реестр региональных ограничений и коммерческих условий, не перегружая ими конституцию.

## Для каждой страны/региона фиксировать
- статус приоритета (primary / secondary / later),
- причины приоритета,
- store availability,
- language requirements,
- privacy/legal constraints,
- tax/commercial notes,
- partner relevance,
- support constraints.

## Текущий принцип
Глобальное распространение с приоритетом по покупательской способности и коммерческой привлекательности.

## Важно
Этот файл — живой реестр. Он меняется чаще конституции и ADR.

## PII inventory (cross-spec — кумулятивно)

Этот раздел перечисляет типы персональных/псевдонимных данных которые приложение собирает, по спекам. Обновляется при добавлении новых типов.

### spec 007: pairing-and-firebase-channel (added 2026-05-11)

| Data type | Classification | Storage | Sensitivity | Retention |
|---|---|---|---|---|
| `managedDeviceId` (UUIDv4) | **Pseudonym** — random ID, не PII в strict GDPR sense | Локально DataStore + Firestore `/devices/{id}` | Low — не привязан к личности | Pока pairing активен + 0 (delete on revoke) |
| Firebase Auth UID (anonymous) | **Pseudonym** — generated by Firebase | Firebase Auth (Google) + Firestore `request.auth.uid` | Low | Пока user не залогаутится |
| `fcmToken` (FCM registration) | **Pseudonym** — per-device ephemeral token | Firestore `/links/{linkId}/state.fcmToken` | Medium — позволяет таргетинг push на конкретное устройство | Регенерируется Google'ом периодически (~weeks/months) |
| `adminId` (anonymous Firebase Auth UID) | **Pseudonym** | Firestore `/links/{linkId}.adminId` | Low | Пока link активен |
| `linkId` (Firestore document ID) | **Pseudonym** — opaque random string | Firestore `/links/{linkId}` | Low | Пока link активен |

**Explicit NOT collected in spec 007**:
- Email, phone, name, address.
- Location (GPS / network).
- Biometric data.
- Contact list (это спек 011).
- Account identifiers (Google, etc.).
- Device IMEI / serial.

### Jurisdiction notes

- **EU (GDPR)**: pseudonymous IDs technically fall under GDPR scope если можно re-link к личности через external data. Текущий design без email/phone/name делает re-identification невозможным в normal use case. На production-релизе — добавить privacy policy с явным disclaimer.
- **Russia (152-ФЗ)**: persistent storage user data in cloud — требуется уведомление Roskomnadzor (если targeting RU users). На production — отдельная задача (TODO для production-release backlog).
- **China**: out-of-scope (Firebase не работает без VPN).
