- причины приоритета,
- store availability,
- language requirements,
- privacy/legal constraints,

- partner relevance,
- support constraints.

### spec 007: pairing-and-firebase-channel (added 2026-05-11; **finalized 2026-05-11** at code-complete)

Cross-checked against shipped code on branch `007-pairing-and-firebase-channel`:
`PairingService`, `FirebaseRemoteSyncBackend`, `FirestoreLinkRegistry`,
`DataStoreDeviceIdProvider`, `LauncherFirebaseMessagingService`.

| Data type | Classification | Storage | Sensitivity | Retention |
|---|---|---|---|---|
| `managedDeviceId` (UUIDv4) | **Pseudonym** — random ID, не PII в GDPR sense | DataStore `com.launcher.pairing.identity_v1` (local) + Firestore `/links/{linkId}.managedDeviceId` | Low — не привязан к личности | До unbind/revoke (delete-on-revoke per FR-033) |
| `managedDeviceFirebaseUid` (anonymous Firebase Auth UID) | **Pseudonym** — generated by Firebase | Firebase Auth + Firestore `/pairings/{token}.managedDeviceFirebaseUid` + `/links/{linkId}.managedDeviceFirebaseUid` | Low | До sign-out / Firestore subtree wipe |
| `adminId` (anonymous Firebase Auth UID) | **Pseudonym** — generated by Firebase | Firebase Auth + Firestore `/pairings/{token}.adminId` (after claim) + `/links/{linkId}.adminId` | Low | До unbind/revoke |
| `linkId` (16-char alphanumeric) | **Pseudonym** — opaque random string generated by `PairingService.generateLinkId` | Firestore `/links/{linkId}` | Low | До unbind/revoke |
| `PairingToken` (`[A-HJ-NP-Z2-9]{6}`) | **Ephemeral pseudonym** — TTL 5 min | Firestore `/pairings/{token}` | Low | 5 min TTL OR delete-on-claim/decline/cancel |
| FCM topic subscription (`link-{linkId}`) | **Pseudonym** | FCM server-side topic map (Google-controlled, not in our Firestore) | Low — opaque to attacker | До `unsubscribeFromTopic` on unbind |
| Per-device FCM registration token | **Pseudonym** — per-device ephemeral | **In-memory on device only** — fetched by `FcmRegistration.currentFcmToken()` for diagnostics; NOT yet persisted to Firestore as of spec 007 (TODO for spec 008 — `LauncherFirebaseMessagingService.onNewToken` plans to write to `/links/{linkId}/state.fcmToken`) | Medium *(future state)* — позволит таргетинг push на конкретное устройство | n/a today; rotating ephemeral when persisted |

**Explicit NOT collected in spec 007**:
- Email, phone, name, address.
- Location (GPS / network).
- Biometric data.
- Contact list (это спек 011).
- Account identifiers (Google, etc.).
- Device IMEI / serial.

### Jurisdiction notes

- **EU (GDPR)**: pseudonymous IDs technically fall under GDPR scope если можно re-link к личности через external data. Текущий design без email/phone/name делает re-identification невозможным в normal use case. На production-релизе — добавить privacy policy с явным disclaimer.
- **Russia (152-ФЗ)**: persistent storage user data in cloud — требуется уведомление Roskomnadzor (если targeting RU users). На production — отдельная задача (TODO для production-release backlog).
- **China**: out-of-scope (Firebase не работает без VPN).
