| CHK017 | Failures aggregated by category for rate measurement | [~] | Implicit via typed errors (`RootKeyError.*`, `BackupError.*`) — categories exist as sealed types. **Gap**: no explicit FR/SC stating «emit metric per error type» for rate-limit measurement (Q-I notes Argon2 as only brute-force defense; server-side rate-limit deferred). Acceptable for MVP given Argon2 work-factor; flag for Phase 5+ when server-side rate-limit added. |

## Summary

**15 PASS, 2 PARTIAL (CHK016, CHK017)** — both about explicit diagnostic-event enumeration. Spec is overwhelmingly recovery-focused (it literally IS the recovery feature), so happy-path-only framing is decisively avoided. Error categories, retry strategies, offline behaviour, and unrecoverable-state UX are explicit throughout FRs, USs, Edge Cases, and Clarifications Q-A through Q-L.

**Recommendation**: do not block. Add a single follow-up FR (or plan-level note) listing diagnostic event categories (`recovery.setup.uploadFailed`, `recovery.entry.wrongPassphrase`, `recovery.fallback.wipeStarted`, etc.) with explicit «no passphrase, no blob material, no email» exclusion rule. Server-side rate-limit aggregation tracked in `docs/dev/server-roadmap.md` per Q-I.
