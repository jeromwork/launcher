## Summary

Ship `simple-launcher` profile as composition of bundled JSON documents on top of the existing F-3 wizard engine (TASK-1 Done). Three architectural gaps in current F-3 implementation must be closed to fulfill constitution Article VII §14 (config-check master) and Article III §7 (stability over system-level changes): (1) engine `computePending(manifest)` pre-flight using `SystemSettingPort.status()` per step, replacing linear traversal; (2) `AppCompatDelegate.setApplicationLocales()` integration for app-level locale override persistence; (3) pool schema v1→v2 bump with declarative `CheckSpec`/`ApplySpec` sealed hierarchies replacing hardcoded `when(settingId)` dispatch.

## Technical Context

**Language/Version**: Kotlin (KMP — commonMain + androidMain). JVM target as per existing project. Compose Multiplatform UI.
**Primary Dependencies**: kotlinx.serialization (existing), Koin (existing DI), AndroidX AppCompat (existing — used for `AppCompatDelegate.setApplicationLocales()`), Compose (existing). **No new external deps.**
**Storage**: DataStore (existing, used by UserPreferencesStore, WizardCheckpointStore, DismissedHintsStore). Android assets for bundled JSON.
**Testing**: kotlin.test + JUnit (commonTest + androidUnitTest), Konsist (architectural fitness functions), Compose UI Test (instrumented).
**Target Platform**: Android API 26+ (existing project floor). Emulator `pixel_5_api_34` for primary local test path (avoiding Compose UI test API 35+ blocker per memory note).
**Project Type**: Mobile app (Android-first KMP; iOS adapters scheduled for TASK-26).
**Performance Goals**:
- Cold-start to first wizard screen ≤ 2s on emulator pixel_5_api_34.
- Wizard exit to HomeActivity first frame ≤ 1s.
- `SystemSettingPort.status()` cached calls < 5ms; uncached < 50ms.
**Constraints**: LOCAL mode (no Sign-In, no cloud network calls). Senior-safe baseline ≥ 56dp / ≥ 24sp / ≥ 4.5:1 contrast (Article VIII §7). APK size delta ≤ +150 KB.
**Scale/Scope**: One profile (`simple-launcher`) shipped. ~6 system settings + ~6 UI options in pools. 3 mandatory + 1 optional wizard steps. 2 new UI surfaces in Settings (pending checklist, walk-through button, locale indicator).

### Data flow — config-check master (Сценарий 2)

```
1. WizardActivity.onCreate()
   ↓
2. ConfigSource.load("wizard-manifest.simple-launcher") → WizardManifest
   ↓
3. engine.computePending(manifest):
   for each StepEntry in manifest.steps:
     when SystemSetting:
       → SystemSettingPort.status(entry.refId)
         → AndroidSystemSettingAdapter.status(settingId)
           → cache.get(settingId, ttl=30s) ?: dispatcher.check(entry.check, params)
             → CheckHandler matching CheckSpec variant class
               → returns SettingStatus
       Applied → exclude from pending
       NotApplied / Indeterminate → include in pending
     when UIChoice:
       → UserPreferencesStore.current()[entry.refId]
       valid value → exclude
       missing/invalid → include
   ↓
4. if pending.isEmpty() → route to HomeActivity (skip wizard)
   else → engine.run(manifest, pendingOnly = pending)
   ↓
5. Wizard renders pending steps sequentially
   For each step:
     - if SystemSetting → SystemSettingPort.applyOrPrompt() → CheckHandler dispatch
     - if UIChoice → user picks value → save to UserPreferencesStore
     - if Custom("pair-admin") → PairAdminCustomStepHandler → launches PairingActivity
   ↓
6. On completion:
   - userPreferencesStore.markWizardCompleted("simple-launcher")
   - AppCompatDelegate.setApplicationLocales(LocaleListCompat.create(...))
   - cache invalidated (lifecycle RESUMED hook fires on return)
   ↓
7. Route to HomeActivity → existing ConfigBackedFlowRepository renders config
```

### Data flow — walk-through mode (FR-014a, Сценарий 5)

```
1. Settings → user taps "Пройти все настройки пошагово" button
   ↓
2. engine.runWalkThrough(manifest):
   Traverses ALL steps (not filtered by computePending)
   For each step:
     - current value loaded from current state (SystemSettingPort.status() or UserPreferences)
     - Display: "Текущее: <current>. [Оставить] [Изменить]"
     - "Оставить" → next step
     - "Изменить" → standard step UI (system dialog, picker, etc) → save
   ↓
3. On completion → route back to Settings
```

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| OEM-specific ROLE_HOME dialogs (Samsung One UI confirm, Xiaomi MIUI quirks) | Medium | Wizard step retry confusing | `SystemSettingPort.status()` re-check on Lifecycle.RESUMED via cache invalidation; inline TODO(physical-device); accept Indeterminate as pending |
| Pool v1→v2 backward-compat read break | Low | App crash on old configs | `PoolSchemaBackwardCompatTest`; gradual migration (v1 entries fall through to legacy dispatch); roundtrip tests |
| `AppCompatDelegate.setApplicationLocales` not behaving uniformly on API < 33 | Medium | Locale doesn't persist on older devices | AppCompat shim handles; fallback to standard Config-based locale; emulator test on API 31 |
| Cache TTL too long → stale status (e.g. user changes ROLE_HOME mid-session) | Low | Misleading "Applied" state | Invalidate-on-resume hook fires on every Activity foreground; 30s TTL conservative |
| Walk-through mode UX complexity creep | Medium | Spec scope balloon | Article II §8 MVP definition: walk-through ships minimal (Оставить / Изменить per step); polish via JSON post-MVP |
| Spec 007 pairing UI subtly broken (orphan dependency) | Low | PairAdmin step fails | Custom step handler catches `LinkRegistry.activate()` failures → toast + Skip; fallback graceful |
| Engine `computePending` introduces silent skip false-positives | Low | User confused why step missing | Indeterminate → include in pending; only Applied excludes; diagnostic emit on every skip |
| Schema bump v1→v2 creates one-way door | High | Future re-design expensive | This is a deliberate one-way door per CLAUDE.md §3; documented in research.md with exit ramp |
