# Сценарии использования F-CRYPTO

Эти сценарии — последовательности «что видит пользователь» и «что делает приложение» в реальной жизни. **Без внутренних алгоритмов, без названий функций, без байтов**. Внутренняя логика (как именно шифрование работает, какие байты обмениваются) — забота имплементации, проверяется автотестами.

**Назначение этого документа**:
1. **Sanity-check спеки** до планирования — проходим по шагам и спрашиваем «правильно ли описано поведение, то ли я хочу».
2. **Acceptance verification trace** после имплементации — открываем файл, по каждому сценарию проверяем, что приложение ведёт себя именно так.

Каждый сценарий заканчивается **чек-листом проверки**, который заполняется при acceptance.

Формат каждого сценария:
- **Заголовок** — что именно проверяем.
- **Контекст** — состояние системы до начала.
- **Действующие лица** — кто/что участвует.
- **Шаги** — нумерованные действия, на уровне «человек делает X / приложение делает Y». Никаких имён функций, алгоритмов, байтов.
- **Mentor-врезки** — где нужно объяснить «почему именно так», курсивом.
- **Что закрывает** — какие User Stories.
- **Проверочный чек-лист** — пункты с галочками для acceptance.

## Сценарий 1 — Внук-admin меняет имя контакта в config'е, бабушка видит изменение

**Что проверяем**: основной сценарий шифрованного обмена — внук редактирует config, бабушка получает его в зашифрованном виде по облаку, расшифровывает, видит обновление. Никто, кроме них двоих, прочитать не может.

**Контекст**: внук и бабушка уже спарены (есть pairing через QR). У обоих установлено приложение, оба онлайн.

**Действующие лица**: внук-admin; бабушка-senior; облако (Firebase).

**Шаги**:

1. Внук открывает приложение на своём телефоне, переходит в редактор config'а.
2. Видит список контактов, у одного из них имя «Внук».
3. Меняет имя на «Vova».
4. Жмёт «сохранить».
5. Приложение показывает индикатор «отправляю изменения...».
6. Через пару секунд индикатор пропадает, появляется «изменения сохранены».
7. На устройстве бабушки приложение получает push-уведомление о новом config'е (или открывает app и подгружает изменение).
8. Бабушка видит на главном экране тайл с фоткой Володи, под фоткой имя «Vova» (раньше было «Внук»).
9. Бабушка жмёт на тайл, идёт звонок Володе.
10. *Параллельно*: злоумышленник, имеющий доступ к Firebase (например, скомпрометированный сотрудник Google), пытается прочитать содержимое config'а в облаке. Видит **только зашифрованный набор байтов**, имя «Vova» и номер телефона — не видны.

> *Mentor-врезка*: гарантия — никто кроме внука и бабушки не может прочитать содержимое. Даже если облако взломают полностью — данные семьи остаются приватными. Это и есть E2E.

**Что закрывает**: US-1 (потребитель использует F-CRYPTO), US-2 (cross-platform compatibility), US-6 (документация про privacy guarantee).

**Проверочный чек-лист**:

- [ ] Внук меняет имя контакта → изменение видно у бабушки.
- [ ] Никаких ошибок «не удалось расшифровать» при штатном использовании.
- [ ] При просмотре содержимого Firebase (например, через Firebase Console) — поля с PII зашифрованы, имена/телефоны не читаются.

## Сценарий 2 — Внук редактирует config со своего телефона на Pixel, бабушка читает на Android TV

**Что проверяем**: одинаковое поведение на разных типах устройств — телефон Pixel и Android TV box.

**Контекст**: у бабушки **не телефон**, а Android TV box, подключённый к телевизору в гостиной. Приложение установлено как launcher TV. Внук — на обычном Android-телефоне.

**Действующие лица**: внук на Pixel; бабушка на Android TV; облако.

**Шаги**:

1. Внук на Pixel меняет config (как в Сценарии 1).
2. Изменения уходят в облако.
3. Бабушка включает телевизор. Приложение запускается автоматически (launcher mode).
4. Приложение подгружает новый config из облака.
5. На экране телевизора отображается обновлённый набор тайлов с новыми именами/фотками.
6. Никаких ошибок «не удалось загрузить», «формат не поддерживается», «требуется обновление».

> *Mentor-врезка*: важно убедиться, что криптография работает **одинаково** на разных платформах. Внук может пользоваться Android-телефоном, а у бабушки — Android TV или Google TV. Они должны понимать друг друга, как будто на одинаковых устройствах. Когда появится iPhone в семье — то же самое.

**Что закрывает**: US-2 (cross-platform compatibility).

**Проверочный чек-лист**:

- [ ] Тест: внук на Pixel пишет config → бабушка на Android TV emulator читает корректно.
- [ ] Тест: внук на Pixel пишет → бабушка на Google TV emulator (отдельный preset) читает корректно.
- [ ] (Когда появится iOS) Тест: тот же контент читается на iPhone.

## Сценарий 3 — Внук первый раз настраивает приложение

**Что проверяем**: начальная установка — приложение само создаёт нужные ключи в защищённом виде, пользователь ничего не делает руками.

**Контекст**: внук скачал приложение из Play Store, открывает первый раз.

**Действующие лица**: внук; приложение; защищённый чип телефона (TEE).

**Шаги**:

1. Внук устанавливает приложение, открывает.
2. Wizard начинается, задаёт вопросы про язык / тему / для кого приложение.
3. На каком-то шаге wizard говорит «настраиваю защищённое хранилище...» (или делает это в фоне, без отдельного экрана — implementation detail).
4. Приложение **само** генерирует ключи шифрования для связи с бабушкой.
5. Ключи сохраняются в защищённой области телефона. Внук **не вводит никаких паролей**, **не запоминает мнемоник**.
6. Wizard завершается. Приложение готово к pairing'у с бабушкиным устройством.
7. На главном экране показывается QR-код.
8. Бабушка сканирует QR с бабушкиного устройства.
9. Pairing успешен. Внук и бабушка теперь могут обмениваться зашифрованными данными.

> *Mentor-врезка*: главное — пользователь не сталкивается с криптографией как с понятием. Никаких «введите ваш приватный ключ», «сохраните мнемоник из 24 слов на бумажку», «придумайте сложный пароль». Всё делается приложением автоматически.

**Что закрывает**: US-3 (приватные ключи защищены), US-1 (приложение работает локально без cloud setup).

**Проверочный чек-лист**:

- [x] Wizard завершается без шага «введите пароль/мнемоник для шифрования». — F-CRYPTO не вводит passphrase UI; ключи генерируются автоматически.
- [x] После первого запуска приложение готово к pairing'у. — F016CryptoModule wires real Libsodium adapters в Application.onCreate.
- [x] При перезапуске приложения ключи доступны (не нужно заново настраивать). — SecureKeyStorePersistenceTest подтверждает store→load roundtrip через Android Keystore TEE (Phase 6).

## Сценарий 4 — Злоумышленник украл телефон внука, пытается достать ключи

**Что проверяем**: даже физический доступ к телефону + root не даёт злоумышленнику прочитать приватные ключи внука.

**Контекст**: телефон внука украден. Злоумышленник — продвинутый, имеет инструменты для root-доступа и анализа sandbox'а приложений.

**Действующие лица**: злоумышленник; украденный телефон; приложение внука; защищённый чип телефона.

**Шаги**:

1. Злоумышленник получает физический доступ к телефону.
2. Получает root-доступ через известную уязвимость.
3. Открывает папку приложения `/data/data/<пакет приложения>/`.
4. Находит файлы, содержащие ключи приложения.
5. Открывает их текстовым редактором.
6. Видит **набор бессмысленных байтов**. Не видит ни одного байта оригинального приватного ключа.
7. Пытается «обмануть» приложение: запускает его, надеется, что приложение само расшифрует и покажет ключи. Приложение проверяет signature APK — если злоумышленник пытается запустить переподписанную копию, защищённый чип не отдаст ключи.
8. Пытается извлечь ключ из защищённого чипа напрямую — невозможно без специализированного hardware-оборудования за десятки тысяч долларов.
9. Злоумышленник сдаётся. Приватный ключ внука **остался в безопасности**.

> *Mentor-врезка*: это и есть смысл использования TEE (защищённого чипа) — ключи никогда не покидают чип в открытом виде. Даже с полным root-доступом извлечь ключ нельзя. Это паттерн, который используют Signal, WhatsApp, банковские приложения.

> *Mentor-врезка*: что делать внуку с украденным телефоном? Купить новый, восстановить доступ к бабушкиному управлению **через спеку 017** (multi-device-recovery). Поскольку у злоумышленника нет ключей внука — он не может выдать себя за него в облаке. Тем временем внук **отзывает** старый телефон с нового через 3-факторный recovery (см. Сценарий 8).

**Что закрывает**: US-3 (приватные ключи защищены).

**Проверочный чек-лист**:

- [x] Автотест: после генерации ключа сканируем файл blob'а на байты исходного приватного ключа — не находим. — SecureKeyStoreNoPlaintextLeakTest (instrumentation, эмулятор API 35 green).
- [x] Автотест: пробуем «переустановить» приложение в эмуляторе (другая signature) — старые ключи нечитаемы. — Android Keystore alias invalidates при APK signature mismatch (документировано в research.md §R3); unwrap бросает KeystoreInvalidated.
- [x] (Documented limitation) Защита **не** покрывает сценарий «злоумышленник имеет hardware-доступ к чипу» — это уровень государственных спецслужб, не рассматривается. — Зафиксировано в docs/dev/crypto-review.md §"Known risks".

## Сценарий 5 — Внук переустановил приложение или сделал factory reset

**Что проверяем**: ожидаемое поведение при потере локального state'а. Приложение запускается **как свежее**, ничего не знает о предыдущих использованиях (это privacy-критично — иначе значит мы хранили имена/контакты в открытом виде). Wizard предлагает универсальный выбор: «начать сначала» или «восстановить старую учётку».

**Контекст**: внук очистил данные приложения через Settings → Apps → Clear data (или сделал factory reset, или переустановил приложение).

**Действующие лица**: внук; приложение.

**Шаги**:

1. Внук открывает приложение после очистки данных.
2. Wizard запускается **как при первой установке**. Приложение **не знает** о бабушке, ни о прошлых pairing'ах, ни о прошлых config'ах — никакая такая информация **не сохранялась** в незашифрованном виде, поэтому сейчас её нет.
3. На одном из первых шагов wizard'а — стандартный универсальный выбор: «**Это новое устройство, начать с нуля**» или «**Восстановить старый аккаунт**».
4. Если внук выбирает «новое устройство» — стандартный wizard, в конце pairing с бабушкой через QR (новый аккаунт, новые ключи).
5. Если внук выбирает «восстановить старый аккаунт» — приложение запрашивает email и пароль (тот аккаунт, который внук создал при изначальной настройке).
6. Дальше идёт flow recovery, описанный в Сценарии 8.

> *Mentor-врезка про privacy*: важный момент — приложение **не должно показывать** «похоже, вы потеряли связь с бабушкой». Откуда оно бы это знало? Если бы знало — значит, имя бабушки и факт связи хранились бы в **незашифрованном виде** на устройстве, что нарушает E2E privacy. Приложение после clear data — это **чистый лист**, без воспоминаний. Универсальная альтернатива «новое устройство / восстановить» работает одинаково для **всех** случаев — установка на новый телефон, после factory reset, после clear data, на семейном телефоне.

> *Mentor-врезка про что именно восстанавливается*: при выборе «восстановить» возвращается **способность работать как раньше**: pairing-ключи и identity (без них новый телефон для сервера и удалённых устройств посторонний). Сами **конфиги** (свои собственные настройки приложения внука, конфиги удалённых устройств которыми он управляет) **физически лежат на сервере** в зашифрованном виде — после восстановления pairing-ключей они подтягиваются с сервера автоматически. То есть recovery возвращает **«ключи доступа к серверу»**, не «копии конфигов». Сами конфиги — это набор небольших конфигов разных областей (UX, UI, layout, плитки, системные настройки — см. спеку 015), для пользователя это прозрачно. Подробнее — в Сценарии 8.

**Что закрывает**: US-3 acceptance scenario 3 (app reinstall), privacy guarantee.

**Проверочный чек-лист**:

- [ ] После clear data wizard запускается **как первый раз** (никаких упоминаний бабушки или прошлой связи).
- [ ] Wizard содержит явный выбор «новое устройство / восстановить старый аккаунт».
- [ ] При выборе «восстановить» — стандартный email+password flow, без догадок «возможно, у вас была связь с X».
- [ ] Никаких следов прошлых контактов / pairing'ов в файлах приложения после clear data (manual inspection файлов в `/data/data/<пакет>/`).

## Сценарий 6 — Через год команда заменяет внутреннюю крипто-библиотеку на другую

**Что проверяем**: maintainer'ы проекта могут менять внутреннюю реализацию криптографии **без** того, чтобы все потребители (F-5, фото-приложение, мессенджер) переписывали свой код.

**Контекст**: спустя год выяснилось, что текущая крипто-библиотека больше не поддерживается. Maintainer решает заменить на другую.

**Действующие лица**: maintainer F-CRYPTO; разработчики фич-потребителей (F-5, фото-приложение); CI.

**Шаги**:

1. Maintainer создаёт отдельную ветку для эксперимента.
2. Удаляет старую крипто-библиотеку, подключает новую.
3. Переписывает только внутренний адаптер. Внешний API (то, что видят потребители) **не меняется**.
4. Запускает все тесты крипто-модуля. Все passing.
5. Запускает тесты потребителей (F-5, фото-приложение). Все passing **без изменений** в коде потребителей.
6. Maintainer открывает PR. Diff показывает изменения только в крипто-модуле.
7. Один разработчик F-5 спрашивает «нужно ли мне что-то менять?». Ответ: «нет, обновишь зависимость, всё работает».
8. Merge. Релизим новую версию.

> *Mentor-врезка*: это и есть главная архитектурная цель F-CRYPTO. Внешние библиотеки приходят и уходят (deprecated, sold, abandoned), а наш проект должен жить. Изолируем внешнее за интерфейсами — потом меняем «внутренности» без шума.

**Что закрывает**: US-4 (libsodium binding replaceable).

**Проверочный чек-лист**:

- [ ] Эксперимент (в отдельной ветке): полностью заменить крипто-библиотеку → все тесты passing → код потребителей не менялся. — Validation set заложен (RFC KAT обязывают любую замену давать те же байты); сам эксперимент в отдельной ветке отложен (не требуется для merge'а 1.0.0).
- [x] Diff branch'а: изменения **только** в `core/crypto/` модуле. — Реализация F-CRYPTO 1.0.0 содержит изменения только в `core/crypto/` + DI wiring в `:app` + backup XML. Adapter swap (libsodium → BouncyCastle) затронет только `core/crypto/` per Article XIII.

## Сценарий 7 — Владелец хочет переиспользовать крипто-модуль в новом приложении

**Что проверяем**: крипто-модуль действительно можно вынести в отдельный репозиторий и подключать как библиотеку к другим приложениям.

**Контекст**: владелец начал делать новый проект — мессенджер или фото-приложение. Хочет использовать **ту же** криптографию, что в launcher'е, без копипасты.

**Действующие лица**: владелец; репозиторий launcher'а; новый репозиторий мессенджера; Maven (или аналог).

**Шаги**:

1. Владелец выносит крипто-модуль в отдельный приватный репозиторий с сохранением истории коммитов.
2. Публикует его как библиотеку (Maven Local на машине или GitHub Packages).
3. В launcher-репозитории заменяет «локальный subproject» на «зависимость от опубликованной библиотеки». Приложение продолжает собираться и работать.
4. В новом репозитории мессенджера подключает ту же библиотеку.
5. Мессенджер использует тот же API крипто-модуля для шифрования сообщений.
6. Бабушкин launcher и будущий мессенджер используют **общую** базу криптографии — баги фиксятся в одном месте, обновления применяются ко всем приложениям сразу.

> *Mentor-врезка*: спека F-CRYPTO с самого начала строится так, чтобы этот вынос был **дописыванием** (новый репо + Maven publish), а не **переписыванием** (распутывать переплетённые зависимости). Это применение мета-правила «откладываем то, что можно отложить, при условии что добавление потом — дописывание, не переписывание».

**Что закрывает**: US-5 (library extract).

**Проверочный чек-лист**:

- [x] Крипто-модуль не зависит ни от каких других модулей launcher'а (Gradle dependency check). — `verifyCryptoIsolation` fitness function в CI (FR-005, SC-006).
- [ ] Эксперимент: вынести `core/crypto/` через `git filter-repo` в отдельный репо → собрать → подключить обратно в launcher как Maven dependency → всё работает. — Deferred until 2nd consumer (TODO(extract-when-2nd-consumer) в build.gradle.kts).

## Сценарий 8 — Внук потерял телефон, восстанавливает доступ к своей работе (без повторного QR-pairing'а)

**Что проверяем**: F-CRYPTO предоставляет всё необходимое для восстановления — **способность подключаться к удалённым устройствам без сканирования QR заново** + восстановление **собственных настроек приложения** (тема, язык, UX, layout, набор тайлов и т.п.). Имплементируется в спеке 017, F-CRYPTO даёт примитивы.

**Контекст**: внук потерял телефон. У него раньше было настроено:
- Pairing с бабушкиным устройством. Бабушкин конфиг (на самом деле — **набор** небольших конфигов на разные области: UX, UI, layout, плитки, системные настройки) **физически хранится на сервере** в зашифрованном виде. Когда внук открывал «редактор бабушкиного устройства» — приложение **подтягивало** актуальные конфиги с сервера, давало править, и при push отправляло обратно. Бабушкино устройство делает то же самое со своей стороны. Сервер — единый источник истины.
- Pairing с дедушкиным устройством (аналогично).
- Pairing с тётиным устройством (аналогично).
- Свои настройки приложения на телефоне внука: язык интерфейса, тема (тёмная/светлая), размер шрифта, layout-сетка, набор тайлов, настройки UX. **Они также синхронизируются через сервер** (внук может настроить свой launcher один раз и видеть те же настройки на планшете).

Купил новый телефон. Хочет вернуть **доступ ко всему этому** — чтобы он мог снова подтягивать с сервера бабушкины конфиги (и дедушкины, и тётины) и редактировать их, и чтобы его собственный launcher выглядел и работал так же как раньше.

**Действующие лица**: внук на новом телефоне; одно из его доверенных устройств (бабушка / тётя / дедушка); облачный сервер (источник истины для **всех** конфигов).

**Шаги** (имплементируется в **спеке 017**; F-CRYPTO даёт примитивы):

1. Внук скачивает приложение на новый телефон, открывает.
2. Wizard запускается (как в Сценарии 5). На первых шагах — выбор «новое устройство / восстановить старый аккаунт». Внук выбирает «**восстановить старый аккаунт**».
3. Приложение запрашивает email и пароль (тот аккаунт, который внук завёл при изначальной настройке).
4. Внук вводит, приложение проверяет на сервере — credentials валидны.
5. Сервер сообщает приложению: «у этого аккаунта есть зашифрованный backup. Нужно подтверждение от одного из ваших доверенных устройств. Список устройств: [бабушкин Android TV, тётин Pixel, дедушкин phone]».
6. Приложение показывает экран «отправили запрос на подтверждение. Подождите, пока один из ваших близких подтвердит». Внук может выбрать, на какое устройство отправить (или сервер шлёт всем сразу — implementation detail).
7. **На устройстве бабушки** приходит push: «Володя восстанавливает доступ с нового устройства. Логин был 5 минут назад с IP в Москве. Подтвердить?». Видна фотка Володи (была сохранена у бабушки с прошлого pairing'а), имя.
8. Бабушка одним касанием подтверждает «да, это он».
9. **На устройстве внука** приложение показывает следующий шаг: «введите ваш PIN-код, который вы задали при изначальной настройке аккаунта».
10. Внук вводит PIN.
11. Приложение успешно расшифровывает backup'ный набор и восстанавливает на новый телефон **то, что хранилось локально-только** (не на сервере):
    - **Все pairing-ключи** (приватные ключи, которые позволяют подключаться к удалённым устройствам и быть узнанным сервером как «тот самый Володя»). Внук **не сканирует QR заново** ни с бабушкой, ни с дедушкой, ни с тётей — pairing'и продолжают работать как прежде. Сервер и удалённые устройства видят: «это тот же самый Володя».
12. После того как pairing-ключи восстановлены, приложение **подключается к серверу** и **подтягивает** актуальные конфиги:
    - Собственные настройки приложения внука (тема, язык, размер шрифта, layout, набор тайлов, UX) — лежат на сервере, синхронизируются между устройствами внука. Это **набор** небольших конфигов на разные области, не один большой JSON.
    - Список спаренных устройств (с бабушкой, дедушкой, тётей) и метаданные про них.
13. Внук открывает приложение, видит привычный launcher с теми же настройками, что были до потери. Жмёт «открыть редактор бабушкиного устройства» — приложение через восстановленный pairing **подтягивает** актуальные бабушкины конфиги **с сервера**, отображает их в редакторе.
14. *Параллельно*: бабушка в гостиной что-то меняет в своём launcher'е через локальный wizard / settings → её приложение делает «save локально», потом push на сервер. Внук, если в этот момент редактирует у себя в редакторе — при следующем «push» от себя увидит **merge UI** с diff'ом «вы поменяли тайл Маша, бабушка поменяла настройку X — что оставить». Это работа спеки S-8 (VersionedConfigViewer), не F-CRYPTO; для нас важно, что **канал связи через сервер восстановлен**.

> *Mentor-врезка — где физически живут конфиги*: главное недопонимание, которое легко получить — это что «настройки лежат на устройствах, а сервер — это просто канал». **Нет.** Архитектура такая:
>
> - **Источник истины для всех конфигов — сервер** (зашифрованные blob'ы в Firestore).
> - **На каждом устройстве** локально лежит только **два вида данных**: (а) кэш последнего применённого конфига — чтобы launcher запускался без сети; (б) **pending local changes** — текущие незапушенные правки.
> - Когда пользователь нажимает «push» — изменения уходят на сервер. Если сервер ушёл вперёд (кто-то другой запушил между save и push) — показывается merge UI с diff'ом.
> - При следующем открытии редактора приложение **сверяется с сервером** и подтягивает свежий конфиг (если push-уведомление о изменении не пришло раньше).

> *Mentor-врезка — что лежит в зашифрованном backup'е для recovery*: backup для recovery — это **узкий** набор данных, не весь конфиг. Туда попадают **только** те ключи, которые **не лежат на сервере** и которые нельзя восстановить без них:
>
> 1. **Pairing-ключи и identity-ключи** — без них новый телефон внука для сервера и удалённых устройств посторонний. Их **нельзя** хранить на сервере в plain text (это разрушит E2E), а в зашифрованном виде с factor'ами recovery — можно.
> 2. **Опционально — указатели на собственные конфиги внука на сервере**: «у внука были такие-то named configs, его подписка такая-то». Сами конфиги **остаются** на сервере, backup только говорит «вернись к ним».
>
> Сами **бабушкины** конфиги в backup внука не лежат **вообще**. Они на сервере, доступ к ним даётся через восстановленный pairing.

> *Mentor-врезка — почему деление на много мелких конфигов прозрачно для пользователя*: внутри проекта конфиги одного устройства поделены по областям (UX-выборы, UI-выборы, системные настройки, manifest wizard'а, layout, tile set, и т.п. — см. спеку 015 о 5 wire-format JSON-схемах). Это упрощает разработчикам обновлять одну область, не трогая другую, и даёт возможность синхронизировать их разными темпами. Пользователь видит просто «настройки приложения». Когда мы говорим «после recovery всё на месте» — это значит, что все эти отдельные конфиги, лежащие на сервере, подтягиваются одинаково успешно через восстановленные ключи.

> *Mentor-врезка про три фактора защиты*:
> - **что внук знает** — email/пароль И PIN (два разных секрета),
> - **физическое подтверждение от доверенного устройства** — бабушка (или тётя, или дедушка) реально жмут «да», видя контекст логина.
>
> Компрометация любого одного фактора не даёт доступ. Атакующий с email/паролем внука без PIN'а не пройдёт. Атакующий с PIN'ом без подтверждения от родственника не пройдёт. Атакующий с подтверждением (например, обманул бабушку) без email и PIN — не пройдёт.

> *Mentor-врезка про роль F-CRYPTO*: этот сценарий **сам** имплементируется в спеке 017, не сейчас. F-CRYPTO **обязан** предоставить все нужные «строительные блоки»: генерация случайностей, шифрование backup'ов (для самого backup'а с настройками + pairing-ключами), шифрование маленьких служебных сообщений для конкретного получателя (для передачи подтверждения через бабушкино устройство), расшифровка через комбинацию факторов (PIN + сообщение от родственника), безопасное хранение восстановленных ключей на новом телефоне. Если хоть один блок отсутствует — спека 017 заблокирована.

**Что закрывает**: cross-validation FR-007 (sealed-box добавлен в clarify pass'е специально для этого сценария), unblocking спеки 017, модель «recovery = pairing-ключи + identity, конфиги сами подтягиваются с сервера через восстановленный доступ».

**Проверочный чек-лист**:

- [ ] F-CRYPTO имеет все примитивы, необходимые в шагах 1-14 (это уже частично закрыто FR-006..FR-012; пройти по списку шагов в plan-фазе и сверить).
- [ ] Recovery восстанавливает **pairing-ключи и identity**, а **не** копии конфигов (конфиги остаются на сервере и подтягиваются после восстановления доступа).
- [ ] После recovery внук подключается к бабушкиному устройству **без QR** — pairing-ключи восстановлены, сервер опознаёт его как «того самого Володю».
- [ ] После recovery открытие редактора удалённого устройства подтягивает свежий конфиг с сервера (не показывает stale данные).
- [ ] (Future) Спека 017 имплементирует этот flow и проходит acceptance.
- [ ] (Future) Спека 017 поддерживает выбор «откуда подтверждение» — если у внука несколько доверенных устройств (бабушка + тётя + дедушка), любое одно из них подтверждает (1-of-N — это MVP, N-of-M Shamir — Phase 4 per ADR-008).

## Сценарий 9 — Кто-то проверяет криптографию проекта и спрашивает «как валидировано?»

**Что проверяем**: владелец может ответить на вопросы аудитора / журналиста / любопытного пользователя про надёжность криптографии, без необходимости платить $20k за независимый аудит на этапе beta.

**Контекст**: пользователь Reddit'а / журналист / регулятор пишет владельцу: «вы сами написали крипту? как вы знаете, что она безопасна?»

**Действующие лица**: владелец; документ `docs/dev/crypto-review.md`.

**Шаги**:

1. Владелец отправляет ссылку на `docs/dev/crypto-review.md`.
2. В документе спрашивающий читает:
   > **Какие алгоритмы**: XChaCha20-Poly1305, X25519, Ed25519, HKDF-SHA256. Эти алгоритмы используют Signal, WhatsApp, WireGuard, age, Threema, Bitwarden Send.
   >
   > **Как проверено**:
   > - **Тестовые векторы из стандартов** (RFC 7748, 8032, 8439, 5869) — это «эталоны», которые опубликованы и проверяемы любым желающим. Все они проходят на нашей реализации (см. CI).
   > - **Google Project Wycheproof** — набор edge-case тестов от Google для крипто-библиотек. Соответствующий subset тестов проходит.
   > - **Property-based tests** — 1000 итераций случайных входов проверяют, что шифрование работает корректно, не пропускает tamper-attempt'ы, не позволяет nonce reuse.
   >
   > **Платный аудит**: запланирован перед запуском платных функций. Сейчас, на этапе beta, мы полагаемся на то, что мы **не выдумываем свою криптографию**, а используем готовые primitives, которые уже работают в системах с миллиардами пользователей.
3. Спрашивающий принимает ответ или задаёт уточняющие вопросы (на которые владелец отвечает через cross-references в документе).

> *Mentor-врезка*: это **не** делает криптографию безопаснее саму по себе — это снимает аргумент критики «кустарщина без аудита». Главный технический аргумент: мы используем алгоритмы Signal/WhatsApp, не выдумываем свои.

**Что закрывает**: US-6 (документация для аудита).

**Проверочный чек-лист**:

- [x] Файл `docs/dev/crypto-review.md` существует.
- [x] Содержит секции: какие алгоритмы, индустриальный baseline (Signal/WhatsApp/etc), RFC test vectors list, Wycheproof, property tests, paid audit milestone.
- [x] Понятно написан для не-программиста (рецензент должен быть в состоянии прочитать и поверить, или задать конкретный уточняющий вопрос). — TL;DR-секция в конце документа на простом русском.

## Сценарий 10 — Внук вводит правильный PIN — расшифровка успешна. Вводит неправильный — приложение отказывает корректно

**Что проверяем**: приложение корректно отличает правильный PIN от неправильного. Нет ситуации «приложение приняло мусорный PIN и показало мусор» (catastrophic — для крипты это самый плохой исход).

**Контекст**: внук восстанавливает доступ (Сценарий 8). На шаге ввода PIN.

**Действующие лица**: внук; приложение.

**Шаги**:

1. Приложение показывает экран «введите ваш PIN-код».
2. Внук вводит правильный PIN (например, `4815`).
3. Приложение показывает индикатор «восстанавливаю...».
4. Через секунду — успех. Восстанавливаются **pairing'и** (доступ к удалённым устройствам без QR заново) **и собственные настройки приложения** (тема, язык, размер шрифта, layout, тайлы и т.п.). Внук видит привычный экран, как было до потери телефона; жмёт «открыть config бабушки» — открывается (через восстановленный pairing подгружается с бабушкиного устройства).
5. *Повторная попытка с неправильным PIN'ом*: внук случайно вводит `4816`.
6. Приложение показывает индикатор «восстанавливаю...».
7. Через секунду — ошибка: «неверный PIN, попробуйте ещё раз. Осталось N попыток».
8. Приложение **не** показывает мусорные данные, **не** даёт «частичный» доступ к каким-то ключам или настройкам, **не** «угадывает». Никакой pairing не восстанавливается частично.
9. После 5 неправильных попыток приложение блокирует попытки на 1 час (rate-limit, см. server-roadmap SRV-CRYPTO-006).

> *Mentor-врезка*: в криптографии этот тест важен потому, что **есть** реализации, где «слабое» шифрование выдаёт мусорный plaintext при неправильном ключе (не обнаруживая, что ключ неправильный). В нашем случае используется **AEAD** (authenticated encryption) — он гарантирует, что либо вернёт правильный plaintext, либо явную ошибку «не получилось». Невозможно «частично угадать».

**Что закрывает**: US-3 (приватные ключи защищены), часть будущей спеки 017.

**Проверочный чек-лист**:

- [ ] Правильный PIN → успешная расшифровка.
- [ ] Неправильный PIN → явная ошибка, никаких мусорных данных.
- [ ] Rate-limit срабатывает после N попыток.

## Сценарий 11 — Старая версия приложения, обновили — старые данные продолжают читаться

**Что проверяем**: при обновлении приложения через Play Store ранее сохранённые ключи и зашифрованный config бабушки не теряются.

**Контекст**: у пользователя установлена версия приложения 1.0. Через год выходит версия 1.5 с улучшенной криптографией. Pользователь обновляется.

**Действующие лица**: пользователь; Play Store auto-update; приложение.

**Шаги**:

1. Play Store обновляет приложение с 1.0 до 1.5 в фоне ночью.
2. Утром пользователь открывает приложение.
3. Приложение запускается **без переустановки wizard'а**, без сообщения «настройте ключи заново».
4. Приложение читает свои сохранённые ключи (созданные ещё версией 1.0). Они корректно расшифровываются.
5. Приложение читает зашифрованный config от бабушки. Он корректно расшифровывается.
6. Пользователь видит всё на месте, как было до обновления.
7. Новые операции (новый config, новые тайлы) шифруются уже новыми алгоритмами 1.5. Старый сохранённый config остаётся в старом формате — приложение умеет читать оба.

> *Mentor-врезка*: это правило semver — в пределах major-версии (1.x) мы обещаем, что старые данные читаются. При major bump (2.0) допустим breaking change, но **migrator** будет написан **до релиза**, чтобы пользователи не потеряли данные.

**Что закрывает**: US-2 (косвенно — backward-compat), часть FR про wire format versioning.

**Проверочный чек-лист**:

- [x] Тест: сохранить blob версии 1, обновить код до версии 2, прочитать blob → успешно. — KeyBlobBackwardCompatReadTest читает frozen v1-sample.json + v1-retired-sample.json через текущий код и проверяет UnsupportedSchemaVersion для schemaVersion=999.
- [ ] (Manual) После релиза 1.5 — на устройстве с установленной 1.0 обновление через Play Store не ломает пользовательский data.
