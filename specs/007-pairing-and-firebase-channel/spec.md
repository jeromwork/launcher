## 0. Зачем этот spec вообще существует *(объяснение для новичка)*

После спеков 003–006 лаунчер работает **локально**: показывает раскладку из mock-конфига, исполняет действия, собирает локальный snapshot провайдеров и здоровья. Но **родственник-admin не может ничего изменить удалённо** — нет канала между его телефоном и телефоном OLD, и **бабушкин телефон не может проснуться** в нужный момент.

Этот спек строит **первый реальный двусторонний канал** между двумя устройствами через Firebase + Cloudflare. Состоит из **четырёх частей**:

1. **Pairing** — admin сканирует QR-код с экрана OLD, получает права на чтение/запись определённых документов в облаке для **этого конкретного OLD-устройства**.
2. **Realtime backend** — реализация `RemoteSyncBackend` (пустой port из спека 003) поверх Firebase: read/write документов, подписка на изменения.
3. **Push-инфраструктура** — Cloudflare Worker + Firebase Cloud Messaging: admin (или сервер) дёргает Worker, Worker отправляет FCM, бабушкин телефон **просыпается в Doze**, лаунчер обрабатывает событие.
4. **Безопасность** — Firestore Security Rules (admin → `/config`, `/commands`; OLD → `/state`, `/capabilities`, `/health`) + Worker authentication (только аутентифицированные клиенты могут заказывать push).

**Что этот спек НЕ делает** (и почему):

- **Не применяет** `/config` к локальной раскладке OLD'а (это спек 008 — bidirectional sync с conflict resolution). Но **показывает в логе** «получен push, прочитан config».
- **Не редактирует** конфиг с admin-UI (это спек 009 — admin mode flows).
- **Не реализует** доменные команды (open Play Store, refresh capabilities) — это спек 009.
- **Не реализует** звонки / Jitsi — это будущий отдельный спек, который **переиспользует** push-сервер из 007.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Pairing через QR (Priority: P1)

Родственник (admin) и пожилой пользователь (OLD) сидят рядом. OLD включает в Settings тумблер «Разрешить удалённое управление», на экране появляется QR-код. Admin со своего телефона (тоже с этим лаунчером в admin-режиме) сканирует QR. После взаимного подтверждения два устройства связаны: admin видит идентификатор OLD'а у себя в списке привязанных, OLD видит на экране consent «Admin <id> получил доступ, чтобы отозвать — выключите тумблер».

**Why this priority**: без pairing вообще ничего удалённого не работает.

**Independent Test**: запустить лаунчер на двух эмуляторах. На одном — войти в admin-mode и нажать «Связать новое устройство». На втором — включить тумблер удалённого управления, показать QR. После сканирования у admin'а появляется новая запись в списке устройств; у OLD появляется consent-экран. Pairing-токен после успешного использования больше не работает.

**Acceptance Scenarios**:

1. **Given** OLD на первом запуске, **When** включает тумблер «Разрешить удалённое управление», **Then** на экране показывается QR-код с deep-link `launcher://pair?token=<6-char>` и таймером 5 минут.
2. **Given** admin сканирует валидный непросроченный токен, **When** делает claim, **Then** в Firestore создаётся `/links/{linkId}` с {adminId, oldDeviceId, createdAt}; `/pairings/{token}.claimed = true`; OLD получает realtime-уведомление.
3. **Given** OLD получил `claimed=true`, **When** показывает consent-экран и пользователь жмёт «Разрешить», **Then** `/links/{linkId}/state` инициализируется; OLD **подписывается на FCM topic `link-{linkId}`**.
4. **Given** токен просрочен, **When** admin пытается claim, **Then** транзакция падает; OLD выдаёт новый токен.
5. **Given** токен уже claimed, **When** второй admin пытается claim, **Then** транзакция падает (atomicity через Firestore transaction).

### User Story 2 — RemoteSyncBackend поверх Firebase (Priority: P1)

Доменные коды лаунчера сегодня видят только `RemoteSyncBackend` (пустой интерфейс из спека 003). После этого спека есть **рабочая реализация** через Firebase Firestore, и `FakeRemoteSyncBackend` для тестов. DI-граф собирает нужную реализацию в зависимости от build flavor.

**Why this priority**: после pairing нужен реальный read/write/listen.

**Independent Test**: в unit-тесте подменить backend на `FakeRemoteSyncBackend`, выполнить базовые операции — все возвращают консистентные результаты. В integration-тесте (требует Firestore emulator) — те же операции против `FirebaseRemoteSyncBackend` дают идентичный observable contract.

**Acceptance Scenarios**:

1. **Given** paired link, **When** OLD пишет `/links/{linkId}/state`, **Then** admin, подписанный на `state`, получает обновление в течение 2 секунд (real Firestore).
2. **Given** admin пишет в `/links/{linkId}/config`, **When** OLD онлайн в foreground, **Then** OLD получает change event через Firestore listener.
3. **Given** OLD offline, **When** admin пишет `/config`, **Then** запись сохраняется; когда OLD возвращается онлайн — получает change event.
4. **Given** Fake в тесте, **When** запись→чтение, **Then** возвращается тот же snapshot.

### User Story 3 — Consent screen и отзыв (Priority: P2)

Когда admin успешно claim'нул токен, OLD показывает consent-экран с adminId и фиксированным списком категорий данных. Пользователь жмёт «Разрешить» или «Отклонить». В Settings всегда виден статус «Привязан к admin <id> с <date> — отвязать».

**Why this priority**: соответствие Article XIV (security & privacy). Без consent — нелегально; без revoke — нарушение CLAUDE.md правила 3 (one-way doors).

**Acceptance Scenarios** (без изменений с предыдущей ревизии):

1. **Given** admin claimed, **When** OLD рендерит consent, **Then** перечислены **все** категории доступа.
2. **Given** «Отклонить», **Then** `/links/{linkId}` удаляется, токен сбрасывается, тумблер выключается, **OLD отписывается от FCM topic**.
3. **Given** успешный pairing, **When** Settings → «Привязка», **Then** виден блок с adminId, датой, кнопкой «Отвязать».
4. **Given** «Отвязать» с двойным confirm, **Then** subtree удаляется, **OLD отписывается от FCM topic**, future-команды отклоняются Security Rules.

### User Story 4 — Push через Cloudflare Worker + FCM (Priority: P1)

*Этот scenario существенно переписан в ревизии 2026-05-11 c.*

Когда admin меняет `/config` или создаёт `/commands/{cmdId}`, бабушкин телефон **просыпается через FCM** и читает изменение — **даже если был в Doze**. Архитектура: admin-приложение **после успешного write в Firestore** вызывает HTTPS endpoint Cloudflare Worker'а. Worker валидирует запрос, берёт Firebase service-account JSON из Cloudflare Secrets, отправляет FCM data-message на topic `link-{linkId}`. OLD получает push через `FirebaseMessagingService`, читает данные из Firestore, обновляет state.

**Why this priority**: без push критическая часть UX не работает — бабушка увидит изменение только когда сама возьмёт телефон в руки (минуты или часы).

**Independent Test**: запустить OLD-эмулятор в Doze (`adb shell dumpsys deviceidle force-idle`); из admin-конца записать `/links/{linkId}/commands/{cmdId}` + дёрнуть Worker; убедиться, что OLD получил FCM data-message в течение 10 секунд.

**Acceptance Scenarios**:

1. **Given** OLD зарегистрирован в FCM с актуальным токеном и подписан на topic `link-{linkId}`, **When** admin пишет `/config`, **Then** admin-приложение вызывает `POST {worker-url}/notify` с payload {linkId, type:"config-changed"}; Worker отправляет FCM data-push; OLD получает push в течение 10 секунд; в логе OLD'а появляется «push received: config-changed».
2. **Given** Worker недоступен (network error), **When** admin пишет `/config` и push fails, **Then** admin видит warning «изменение сохранено, push может задержаться»; OLD получит изменение через Firestore listener при возврате online.
3. **Given** OLD-устройство **без GMS** (китайская прошивка), **When** OLD запускается, **Then** FCM-регистрация падает gracefully; WorkManager стартует periodic polling 15 минут; в Settings баннер «уведомления недоступны, обновления раз в 15 минут».
4. **Given** OLD заснул на 30 минут с подпиской на topic, **When** admin шлёт push, **Then** Android доставляет в течение 10 секунд (high-priority data-message с `priority: "high"` в FCM payload).
5. **Given** FCM-токен OLD'а ротирован Google (стандартная политика), **When** OLD выходит онлайн, **Then** новый токен пишется в `/links/{linkId}/state.fcmToken`; old token expires в FCM автоматически.

### User Story 5 — FakeRemoteSyncBackend для разработки и тестов (Priority: P2)

Все доменные тесты и UI-тесты используют `FakeRemoteSyncBackend` (in-memory, deterministic). Build flavor `mockBackend` собирает приложение без Firebase, **без Worker'а** — push'и идут через FakePushSender (no-op, увеличивает счётчик). Это позволяет запускать тесты на CI без Firebase emulator и без Cloudflare account.

**Acceptance Scenarios**:

1. **Given** unit test использует `FakeRemoteSyncBackend`, **When** запись→чтение, **Then** возвращается тот же snapshot.
2. **Given** Fake в `mockBackend` build, **When** оба «устройства» в одном процессе пишут друг другу, **Then** listener'ы получают изменения синхронно.
3. **Given** test использует `FakePushSender`, **When** домен вызывает `pushSender.notify(linkId, type)`, **Then** sender'у инкрементится счётчик, в логе теста — лог-строчка; никаких сетевых вызовов.
4. **Given** test использует Fake, **When** `fake.setOnline(false)`, **Then** writes идут в queue; reads возвращают last-known-good с `isStale=true`; observers не получают updates до `setOnline(true)`.

### Edge Cases

- Pairing-токен пытаются использовать **дважды** одновременно (race) — Firestore transaction обеспечивает atomic claim.
- Pairing-токен **просрочен** (>5 мин) — клиент удаляет при чтении.
- Admin **второй раз** сканирует тот же QR — UX-ветка «вы уже связаны».
- **OLD без GMS** — fallback на WorkManager polling 15 мин (см. US-4 scenario 3).
- **Network** падает прямо во время pairing transaction — admin видит ошибку, может ретраить.
- `google-services.json` **отсутствует** (mockBackend flavor) — приложение собирается, использует Fake.
- Admin **закрыл** приложение между «scan QR» и consent OLD'а — записи созданы; если OLD отклоняет — всё удаляется.
- Пользователь **отвязал** устройство, но admin читает кэш — Security Rules отказывают на следующем чтении.
- **Cloudflare Worker недоступен** — admin видит warning, Firestore listener дойдёт без push.
- **FCM token rotation** Google'ом — OLD пере-пишет токен в `/state.fcmToken` через `onNewToken()` callback.
- **Race**: admin пишет `/config`, дёргает Worker, OLD уже подписан на старый topic из cancelled-pairing — Worker шлёт на текущий topic, проблем нет (topics — короткоживущие).
- **Worker лимит** — Cloudflare 100k req/day; при превышении (теоретически невозможно для MVP) — клиент получает 429, retry с exponential backoff.

## Requirements *(mandatory)*

### Functional Requirements

**Pairing**

- **FR-001**: Лаунчер MUST генерировать `oldDeviceId` (UUIDv4) при первом запуске и хранить в DataStore.
- **FR-002**: Лаунчер MUST регистрироваться в Firebase Auth как anonymous при первом запуске (realBackend flavor).
- **FR-003**: При включении тумблера лаунчер MUST создавать `/pairings/{token}` с `{token, oldDeviceId, claimed:false, expiresAt:now+5min}`, `token` ∈ `[A-HJ-NP-Z2-9]{6}`.
- **FR-004**: Лаунчер MUST показывать `token` как QR с deep-link `launcher://pair?token=<token>` + countdown.
- **FR-005**: Admin-режим MUST читать QR через камеру (CameraX + ML Kit Barcode).
- **FR-006**: Admin MUST выполнять «claim» как **Firestore transaction**: проверить exists, !claimed, !expired; атомарно установить claimed=true; создать `/links/{linkId}`.
- **FR-007**: OLD MUST слушать `/pairings/{token}` и при `claimed=true` показывать consent-экран с adminId + фиксированный список категорий.
- **FR-008**: При «Отклонить» лаунчер MUST удалить `/links/{linkId}` и **полностью удалить `/pairings/{token}`** (single-use).
- **FR-009**: При «Разрешить» лаунчер MUST записать initial snapshot в `/links/{linkId}/state` + **подписаться на FCM topic `link-{linkId}`** (см. push секцию).

**RemoteSyncBackend**

- **FR-010**: Лаунчер MUST предоставлять интерфейс `RemoteSyncBackend` в `:core/api/sync`: `writeDoc(path, snapshot)`, `readDoc(path)`, `observe(path): Flow<DocSnapshot>`, `runTransaction(block)`, `dispose()`.
- **FR-011**: Лаунчер MUST содержать `FirebaseRemoteSyncBackend` — Firestore-adapter (androidMain, realBackend).
- **FR-012**: Лаунчер MUST содержать `FakeRemoteSyncBackend` — in-memory с queue/isStale offline-симуляцией (см. C5).
- **FR-013**: Firebase-adapter MUST **никогда** не пропускать наружу типы Firebase SDK (anti-corruption layer per CLAUDE.md rule 2).
- **FR-014**: Все доменные функции MUST принимать `RemoteSyncBackend` через DI; никаких прямых вызовов Firebase в домене.

**FCM Receiver (на стороне OLD)**

- **FR-015**: Лаунчер MUST регистрироваться в FCM при наличии Google Play Services; полученный `fcmToken` пишется в `/links/{linkId}/state.fcmToken`.
- **FR-016**: Лаунчер MUST содержать `FirebaseMessagingService` (`androidMain`, realBackend flavor) который обрабатывает **data-only silent push** с типами `config-changed`, `command-issued`, `revoke`. На приём push — лаунчер читает Firestore, обновляет state, **не показывает системное уведомление** (silent).
- **FR-017**: При `onNewToken()` (ротация Google'ом) лаунчер MUST обновить `/links/{linkId}/state.fcmToken`.
- **FR-018**: Если GMS недоступны, лаунчер MUST graceful-fallback на WorkManager periodic polling 15 минут; в Settings — баннер.

**Push Server (Cloudflare Worker)** *[новая секция]*

- **FR-019**: Проект MUST содержать каталог `push-worker/` в корне репо: исходник Worker'а (TypeScript ~50 LOC) + `wrangler.toml` (Cloudflare deploy config) + README с инструкцией deploy.
- **FR-020**: Worker MUST экспортировать единственный endpoint `POST /notify` принимающий JSON `{schemaVersion:1, linkId: string, type: "config-changed"|"command-issued"|"revoke", payload?: JsonElement}`. Контракт — `contracts/worker-notify.md` (создаётся в `/speckit.plan`).
- **FR-021**: Worker MUST валидировать запрос через **Firebase ID-token** в `Authorization: Bearer <token>` header: использовать Firebase Admin SDK (или JOSE-библиотеку с публичными ключами Firebase) для проверки подписи. Невалидный токен → 401.
- **FR-022**: Worker MUST хранить **Firebase service-account JSON** в Cloudflare Secrets (`wrangler secret put FIREBASE_SA_JSON`); secret **никогда не попадает в репо**.
- **FR-023**: Worker MUST отправлять FCM data-message на topic `link-{linkId}` через **FCM HTTP v1 API** (`https://fcm.googleapis.com/v1/projects/{projectId}/messages:send`) с OAuth access-token, полученным из service-account через RS256 JWT.
- **FR-024**: Push payload (от Worker'а в FCM) MUST иметь `priority: "high"`, `data: {schemaVersion:"1", type, linkId, ...}`, no `notification` field (silent push). Контракт — `contracts/fcm-payload.md`.
- **FR-025**: Worker MUST содержать минимальный rate-limit (например 100 req/min на linkId) против abuse. Реализация — Cloudflare KV (опционально) или in-memory cache (для MVP — достаточно).

**Firestore schema & security**

- **FR-026**: Wire-format `/pairings/{token}` MUST содержать `schemaVersion: Int` (=1).
- **FR-027**: Wire-format `/links/{linkId}/state` MUST содержать `schemaVersion: Int` (=1); поля включают `fcmToken: String?`.
- **FR-028**: Firestore Security Rules MUST разрешать:
  - admin (`request.auth.uid == link.adminId`): write `/links/{linkId}/config`, `/links/{linkId}/commands/*`; read `/links/{linkId}/`.
  - OLD (`request.auth.uid == link.oldDeviceId-mapped-uid`): write `/links/{linkId}/state`, `/capabilities`, `/health`, `/commands/{cmdId}/result`; read `/links/{linkId}/`.
  - сторонние uid: отказ.
- **FR-029**: Security Rules MUST разрешать создание `/pairings/{token}` только anonymous-authenticated пользователям (anti-spam).
- **FR-030**: Все записи MUST содержать `updatedAt: Timestamp` (server-side `FieldValue.serverTimestamp()`).

**Consent & revoke**

- **FR-031**: Лаунчер MUST показывать в Settings блок «Привязка к admin» с adminId, датой, «Отвязать».
- **FR-032**: «Отвязать» MUST требовать второго подтверждения.
- **FR-033**: После «Отвязать» лаунчер MUST **hard-delete** `/links/{linkId}` со всем subtree; **отписаться от FCM topic `link-{linkId}`**; переключить тумблер в выключенное.

**Build flavor & DI**

- **FR-034**: Проект MUST иметь два flavor'а: `realBackend` (Firebase + FCM + Worker calls) и `mockBackend` (Fake + FakePushSender).
- **FR-035**: Koin граф MUST подменять `RemoteSyncBackend` и `PushSender` в зависимости от flavor через source-set discovery.

**End-to-end consumer (`config-changed`)** *[per C14]*

- **FR-036**: Admin-режим лаунчера MUST после успешного write в `/links/{linkId}/config` вызвать `PushSender.notify(linkId, type="config-changed")`. На стороне `WorkerPushSender` это HTTPS `POST /notify` с Firebase ID-token; на стороне `FakePushSender` — increment counter (для тестов).
- **FR-037**: OLD при приёме FCM data-message с `type="config-changed"` MUST: (а) прочитать `/links/{linkId}/config` через `RemoteSyncBackend.readDoc()`; (б) залогировать в Android Logcat «config received: schemaVersion=N, updatedAt=...». **Применение конфига к UI — out-of-scope** (это спек 008). TODO в коде `PushReceiver.kt` — «expand here in spec 008: apply config to local Room database + trigger UI refresh».

### Key Entities

- **PairingToken**: 6-символьный alphanumeric, TTL 5 мин, single-use.
- **Link**: `{linkId, adminId, oldDeviceId, createdAt}`.
- **DocPath**: доменная обёртка над path в Firestore.
- **DocSnapshot**: `{data: JsonElement, schemaVersion, updatedAt, isStale}`.
- **RemoteSyncBackend**: port для read/write/observe Firestore docs.
- **PushSender**: port для «отправь push с этим payload на этот linkId». Две реализации: `WorkerPushSender` (HTTPS call к Cloudflare Worker'у), `FakePushSender` (counter).
- **PushReceiver**: domain handler входящих push'ей; mapping push-type → действие.
- **AdminIdentity** / **OldIdentity**: anonymous Firebase Auth UID.

- Worker — stateless (или с минимальным KV для rate-limiting), нет database.
