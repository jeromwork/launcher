# UX Checklist: WhatsApp Contact Tiles via Communication Shell

**Purpose**: Проверка качества UX-требований в спецификации (полнота, ясность, согласованность, измеримость), а не проверка реализации.
**Created**: 2026-04-02
**Feature**: [spec.md](../spec.md)

**Note**: Чеклист проверяет, насколько хорошо написаны требования к UX для Feature 002.

## Полнота требований

- [ ] CHK001 Определены ли UX-требования для всех ключевых экранов потока (плитка, экран подтверждения, возврат в Launcher)? [Completeness, Spec §In Scope, Spec §User Scenarios]
- [да ] CHK002 Зафиксированы ли требования к визуальной структуре плитки (верх: фото+имя, низ: две кнопки) без пропусков по элементам? [Completeness, Spec §Clarifications, Spec §FR-002, Spec §FR-003]
- [ ] CHK003 Указано ли, какие UX-состояния обязательны при недоступном переходе в WhatsApp? [Completeness, Spec §FR-010]
- [ ] CHK004 Определены ли требования к UX при отмене действия до handoff, включая ожидаемое состояние домашнего экрана? [Completeness, Spec §FR-009, Spec §User Story 3]

## Ясность требований

- [ ] CHK005 Однозначно ли различаются термины "contact tile", "action confirmation screen" и "return context" во всех разделах? [Clarity, Spec §Key Entities, Spec §FR-004, Spec §FR-007]
- [ ] CHK006 Ясно ли сформулировано, что действия в плитке строго ограничены кнопками `Call` и `Video` (без скрытых альтернатив)? [Clarity, Spec §FR-003, Spec §Clarifications]
- [ ] CHK007 Уточнено ли в требованиях, какой именно "expected state" должен быть восстановлен после возврата? [Ambiguity, Spec §User Story 2, Spec §FR-008]
- [ ] CHK008 Исключены ли расплывчатые UX-формулировки без критериев (например, "predictable") или они операционализированы? [Ambiguity, Spec §SC-004]

## Согласованность требований

- [ ] CHK009 Согласованы ли между собой In Scope и Functional Requirements по структуре увеличенной плитки и действиям `Call`/`Video`? [Consistency, Spec §In Scope, Spec §FR-002, Spec §FR-003]
- [ ] CHK010 Нет ли конфликта между требованием явного подтверждения и сценариями запуска действий с плитки? [Consistency, Spec §FR-004, Spec §FR-006, Spec §User Story 1]
- [ ] CHK011 Согласованы ли edge-case требования по множественным нажатиям с защитой от дублей запуска? [Consistency, Spec §Edge Cases, Spec §FR-011]

## Качество критериев приемки

- [ ] CHK012 Можно ли объективно оценить UX-успех для каждого ключевого шага пути пользователя, а не только для итогового перехода? [Acceptance Criteria, Spec §SC-001, Spec §SC-004, Gap]
- [ ] CHK013 Привязаны ли метрики времени и успешности к конкретным UX-моментам (подтверждение, переход, возврат) без двусмысленности? [Measurability, Spec §SC-002, Spec §SC-003]
- [ ] CHK014 Указаны ли качественные UX-критерии для различения успешного и проблемного восстановления состояния после возврата? [Gap, Spec §SC-003, Spec §FR-008]

## Покрытие сценариев

- [ ] CHK015 Полностью ли описаны альтернативные UX-сценарии для обоих действий (`Call`, `Video`) на экране подтверждения? [Coverage, Spec §User Story 1, Spec §FR-003, Spec §FR-004]
- [ ] CHK016 Описано ли требуемое UX-поведение при возврате разными способами (Back/Home/app switcher) без пробелов? [Coverage, Spec §Edge Cases, Spec §FR-012]
- [ ] CHK017 Определены ли UX-требования для длинной паузы между уходом в WhatsApp и возвратом в Launcher? [Coverage, Spec §Edge Cases, Spec §FR-008]

## Нефункциональные UX-аспекты

- [ ] CHK018 Зафиксированы ли требования по доступности (контраст, размер зоны нажатия, читаемость имени контакта) или это явно исключено? [Gap, Non-Functional]
- [ ] CHK019 Зафиксированы ли требования по локализации UX-текстов для кнопок и подтверждения или это явно отложено? [Gap, Non-Functional]
- [ ] CHK020 Определены ли наблюдаемые UX-сигналы для диагностики проблем возврата контекста без тяжелых фоновый наблюдателей? [Non-Functional, Spec §Constraints, Spec §FR-010]

## Зависимости и допущения

- [ ] CHK022 Явно ли отражено влияние mock-конфигурации на UX (возможные ограничения контента плитки/действий)? [Assumption, Spec §FR-013, Spec §Assumptions]
