# Meta Minimization Checklist: WhatsApp Contact Tiles via Communication Shell

**Purpose**: Проверить, что требования формулируют минимально необходимый набор действий, данных и разрешений для безопасной handoff-последовательности без лишней сложности.
**Created**: 2026-04-02
**Feature**: [spec.md](../spec.md)

**Note**: Чеклист оценивает качество требований (ограниченность scope, минимизация данных, компактность UX, ограниченное взаимодействие с внешним мессенджером).

## Scope Minimalism

- [ ] CHK001 Указано ли, что плитка может инициировать только одно действие за раз (Call или Video) и не допускается расширенный список действий? [Completeness, Spec §In Scope, Spec §FR-003]
- [ ] CHK002 Подтверждено ли, что UI плитки ограничен фото+имя сверху и двумя кнопками снизу (нет дополнительных элементов, индикаторов или меню)? [Clarity, Spec §Clarifications, Spec §FR-002]
- [ ] CHK003 Описано ли, что confirmation flow не требует дополнительных шагов помимо выбора действия и подтверждения? [Completeness, Spec §User Story 1, Spec §FR-004]

## Data Minimalism

- [ ] CHK004 Уточнено ли, какие именно данные хранятся в `Return Context`, и удалены ли все лишние поля? [Clarity, Spec §Key Entities, Spec §FR-007]
- [ ] CHK005 Записано ли, что `Return Context` используется только ради восстановления домашнего экрана и не хранит содержимое сообщений или прочую чувствительную информацию? [Ambiguity, Spec §Key Entities, Spec §FR-007]
- [ ] CHK006 Есть ли требование, ограничивающее количество сохранённых слоёв состояния (например, только текущая плитка), чтобы не раздувать контекст? [Gap, Spec §FR-007, Spec §User Story 2]

## Permission & Integration Minimalism

- [ ] CHK007 Очерчено ли, какие минимальные разрешения требуются от Launcher и вызываются ли они только во время подтверждённого handoff? [Completeness, Spec §Constraints, Spec §FR-006]
- [ ] CHK008 Подтверждена ли обязанность Launcher не расширять API/хендшифтов WhatsApp сверх обоснованного handoff (чтобы избежать full cross-app control)? [Constraint, Spec §In Scope, Spec §FR-014]
- [ ] CHK009 Описана ли минималистичная стратегия реакции, когда WhatsApp недоступен (откат в Launcher без дополнительных опций)? [Coverage, Spec §FR-010, Spec §Edge Cases]

## Edge-Case Minimalism

- [ ] CHK010 Уточнено ли поведение при повторных нажатиях, чтобы не запускалось несколько handoff-ов и UI не порождал лишние переходы? [Consistency, Spec §Edge Cases, Spec §FR-011]
- [ ] CHK011 Описано ли, что возврат в Launcher происходит только с нужным контекстом, без дополнительных шагов согласования контекста? [Consistency, Spec §User Story 2, Spec §FR-008]
- [ ] CHK012 Проверено ли, что out-of-scope зоны (embedded UI, device readiness, background tracking) окончательно исключены, чтобы не вводить не минимальные требования? [Gap, Spec §Out of Scope, Spec §FR-014]

## Behavioral Minimalism

- [ ] CHK013 Прописан ли простой UX (из плитки на экран подтверждения и обратно) без ветвлений или дополнительных состояний? [Clarity, Spec §User Scenarios, Spec §FR-004]
- [ ] CHK014 Описано ли, что mock-конфигурация не требует множественных отдельных опций, а всегда предоставляет минимально достаточный набор контакт+действие? [Traceability, Spec §FR-013, Spec §Assumptions]
- [ ] CHK015 Есть ли требование, что возвращение из WhatsApp не вызывает сложных пересмотров UX (не требуется дополнительного согласования/настройки)? [Coverage, Spec §FR-008, Spec §Edge Cases]
