## Зачем F-014.2 нужна

F-014.1 хранит admin's named configs в Firestore **в plain JSON**. Это значит:
- Google (Firebase staff) может прочитать configs (contact references, document references, tile arrangements).
- При breach Firestore — leaked.
- Privacy regression vs F-014.0 (где все данные на устройстве пользователя).

F-014.2 шифрует ConfigDocument **до** push'а на сервер, через mechanism который определён в **спеке F-5**.

---

## Что входит в F-014.2 scope

### Адаптация под F-5 encryption mechanism

F-5 spec ещё не написана, но из roadmap.md следует:
- Client-side encryption ConfigDocument содержимого перед write в Firestore.
- Recipient model — admin's own UID + paired Managed UIDs (для cross-device sync admin + cross-device target editing бабушки).
- Key management через спеку 011 envelope encryption pattern (libsodium-backed).

### F-014.2 conrete tasks (когда F-5 ready):

- Wrap `RemoteNamedConfigsStore.write()` call в encryption pass.
- Wrap `RemoteNamedConfigsStore.observe()` Flow в decryption pass.
- Handle key rotation events (когда admin rotates его keypair).
- Handle "cannot decrypt" gracefully — UI fallback "Конфиг повреждён, нужно вытащить из cache" + retry.
- Migration F-014.1 → F-014.2: existing plain configs нужно перешифровать.

### No new wire format

Encryption — это transport-level concern (envelope around ConfigDocument).
ConfigDocument schema **не меняется** в F-014.2.

## Open questions для F-014.2

1. **Migration strategy F-014.1 → F-014.2**:
   - Background re-encrypt of all existing plain configs?
   - One-shot на app update?
   - Lazy (encrypt only at next write)?

2. **Key rotation impact**:
   - Admin rotates keypair (F-5 mechanism).
   - All historical configs нужно re-wrap envelope под new key?
   - Cost = N configs × M devices.

3. **Target named configs (FR-003h)**:
   - Бабушкин config — `ownerUid = admin Google UID`.
   - Бабушка не имеет Google account, но её устройство нужно decrypt configs.
   - Recipient model — admin device + paired Managed device.
   - Это работает per envelope pattern (spec 011) — нужно verify.

4. **Performance impact**:
   - Encrypt/decrypt per write/read latency.
   - Background sync — приемлемо.
   - User-facing (push edit dialog → encrypt → push) — может быть 100-300ms задержка.
