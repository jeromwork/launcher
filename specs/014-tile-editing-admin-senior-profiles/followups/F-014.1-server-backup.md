## Зачем F-014.1 нужна

F-014.0 хранит named configs **только локально** в DataStore. Это значит:
- Admin не может cross-device sync configs (один телефон vs планшет).
- При reset / re-install — все configs теряются.
- Multi-device admin (FR-003) — design ready но **не функционален**.

F-014.1 добавляет **Firestore backup** для admin's named configs + cross-device sync per FR-003b/c/d/e/f/g/h/i.

## Что входит в F-014.1 scope

### Functional Requirements уже specified в spec.md, но deferred:

- **FR-003a (full)** — atomic single-default invariant через Firestore transaction (currently local-only, atomicity guaranteed by DataStore.edit, но cross-device race не handled).
- **FR-003b (full)** — orphan UI marker rendering через My Configs screen.
- **FR-003c (full)** — 5-config limit + prompt «Удалить самый старый orphan» через Firestore atomic update.
- **FR-003d (full)** — progressive disclosure UI: My Configs entry в Settings, config switcher, push edit dialog.
- **FR-003e** — push edit dialog logic per state (single+single → silent, single+multi → choice, multi+multi → choice with multi-device note).
- **FR-003f** — My Configs screen UI (list of configs + actions per row).
- **FR-003g** — anonymous → Google migration dialog (3 options: replace / save local as new / skip backup).
- **FR-003h** — target named configs ownership = admin Google UID.
- **FR-003i** — compatibility check при first-install + Google Sign-In success.

### Wire format changes:

- **ConfigDocument schema bump v1 → v2**: добавить fields `configName`, `description`, `isDefault`, `presetId`, `deviceClass`, `activeDeviceIds`, `orphanedAt`.
- Backward-compat: v1 reads → migrate to v2 на client side при save.
- Forward-compat: v3 (future) reads → fail-closed per контракту.

### New adapter:

- `RemoteNamedConfigsStore` port в `core/commonMain/api/edit/`.
- `FirestoreRemoteNamedConfigsStore` adapter в `core/androidMain/adapters/edit/`.
- `MergedNamedConfigsRepository` (commonMain) — composes local + remote stores, handles merge conflicts.
- `FakeRemoteNamedConfigsStore` для tests.

### New UI:

- My Configs screen (Settings → «Мои конфиги (N/5)»).
- Push edit dialog (3 variants per FR-003e).
- Anonymous → Google migration dialog (3 options per FR-003g).
- Subtle toast на first multi-config creation (FR-003d Transition 0→2).

### Firestore Security Rules:

- `/admin-self-configs/{adminUid}/configs/{configName}/current` — write only by `request.auth.uid == adminUid`.
- Single-default invariant — enforced через transaction (Cloud Functions free tier disabled → all enforcement client-side + Rules).

## Research questions open

См. [TODO-FUTURE-RESEARCH-011](../../docs/dev/project-backlog.md) backlog entry:

1. **Concurrent edit merge UI/UX** — что показывать когда:
   - Admin device A edited "home" config, admin device B simultaneously edited "home" config.
   - First push wins (optimistic concurrency), но user на B видит "Конфликт. Обновить?" — какой UI design?
   - Это US-2 для F-014.1, не покрыт в F-014.0 (там single-device).

2. **Multi-device device identity** — TODO-RESEARCH-009:
   - Firebase Installations (FID) per device — ✅ stable across app reinstall если Auto Backup.
   - Но: Auto Backup не enabled для launcher app (HOME role restriction)?
   - Need explicit reset path.

3. **Migration UX** — TODO-RESEARCH-010:
   - User installed F-014.0 локально, накопил configs.
   - Update to F-014.1 + Google Sign-In: prompt'имся на migration?
   - 3 options (replace / save local as new / skip backup) — какой default?
   - Что если user отказался — что блокирует sign-in retry?

## TL;DR на русском

**Почему отложено**: blocked by F-4 (нужен stable Google UID), TODO-RESEARCH-009 (device identity), TODO-RESEARCH-010 (migration UX). Эти blockers — independent работа.

**Effort estimate** (когда blockers closed): 7-10 дней + 3-4 дня на research + UX iteration. Включает полный clarify/plan/tasks/implement/smoke цикл.

**Когда start**: после того как F-4 ships и оба research items closed.
